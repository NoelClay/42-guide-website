# 제0장. 선수 개념 — MBR 문서를 읽기 전 반드시 알아야 할 것들

> **이 장의 목적**: Wikipedia 의 Master Boot Record 문서는 "중학생도 읽을 수 있게" 쓰여 있지 않다. 한 문장에 비트, 바이트, 16진수, little-endian, 섹터, LBA, 레지스터, real mode 같은 개념이 6~7개씩 녹아 있다. 이 장은 **그 개념들을 처음부터 수학적으로 쌓아 올려** 본문을 읽을 수 있게 해주는 기초 공사다. 이 장을 건너뛰고 제1장으로 가면 벽에 부딪힌다.

---

## 왜 이 장이 필요한가 — 한 문장 해부

Wikipedia MBR 문서의 실제 문장:

> *"The MBR signature will appear in a disk editor as the sequence 55 AA. On x86 compatible machines, this can be written as 16-bit word AA55 in programs for x86 processors (note the swapped order), whereas it would have to be written as 55AA in programs for other CPU architectures using a big-endian representation."*

이 한 문장에 숨어있는 전제들:

| 전제 개념 | 이 장의 섹션 |
|---|---|
| **byte** 가 무엇이고 왜 8 bits 인가 | §1 |
| **55, AA** 라는 숫자는 16진수(hex)이다 | §2 |
| **x86** 은 CPU 아키텍처의 이름이다 | §7, §8 |
| **16-bit word** = 2 byte 덩어리 | §1, §2 |
| **swapped order** = little-endian 이라는 저장 방식 | §3 |
| **big-endian** = 반대편 저장 방식 | §3 |
| **disk editor** = 디스크 원시 데이터를 섹터 단위로 보여주는 도구 | §4 |

일곱 전제가 머리에 들어와야 이 한 문장이 읽힌다. 지금부터 하나씩 쌓는다.

## §1. bit 과 byte — 컴퓨터의 가장 작은 알갱이

### 1-1. 컴퓨터는 0 과 1 만 안다

컴퓨터 내부에서 정보를 나르는 전선은 **전기가 흐르거나 / 안 흐르거나** 둘 중 하나다. 이 두 상태에 숫자를 매긴 것이 **0 과 1** 이다.

- 전기 없음 → **0**
- 전기 있음 → **1**

이 하나짜리 0-또는-1 을 **bit** (binary digit, 이진 숫자의 줄임말)라고 한다.

### 1-2. bit 하나로는 아무것도 못한다

bit 하나는 "예/아니오" 딱 두 가지만 표현 가능하다. "사과"나 "3.14"나 "안녕하세요"를 표현하려면 bit 을 여러 개 묶어야 한다.

- 2개 묶으면: 00, 01, 10, 11 → 4가지 (= 2²)
- 3개 묶으면: 000~111 → 8가지 (= 2³)
- 4개 묶으면: 0000~1111 → 16가지 (= 2⁴)
- **8개 묶으면**: 00000000~11111111 → **256가지** (= 2⁸)

이 "8개 묶음" 이 바로 **byte** 다.

### 1-3. 왜 하필 8 bits 가 1 byte 인가

역사적 이유다. 1960년대 IBM System/360 컴퓨터가 문자 하나를 저장하는 데 8 bits 를 할당하는 설계를 채택했고[^bcdic], 이게 산업 표준이 되어 지금까지 왔다. 8 bits 에는 특별한 물리적 이유가 있는 게 아니라 **역사적 합의**다.

[^bcdic]: **BCDIC 과 EBCDIC** — 1960년대 IBM 은 6-bit 문자 코드(BCDIC) 를 쓰다가 8-bit 코드(EBCDIC) 로 확장했다. System/360 (1964) 이후 8-bit = 1 byte 가 고착화되었다. 그 전엔 7-bit byte 도 있었다.

### 1-4. byte 한 개의 표현력

1 byte = 8 bits 이므로 표현 가능한 숫자 개수 = 2⁸ = 256. 값 범위 = **0 ~ 255**.

MBR 에 나오는 모든 숫자(0x80, 0xAA, 0x55 등)는 byte 단위로 읽으면 이 범위 안에 들어간다.

### 1-5. 큰 숫자는 byte 여러 개로

1 byte 로 모자라는 큰 숫자는 byte 를 여러 개 묶어서 표현한다.

| 묶음 이름 | byte 개수 | bit 개수 | 표현 범위 |
|---|---|---|---|
| byte | 1 | 8 | 0 ~ 255 = 2⁸ − 1 |
| word (16-bit) | 2 | 16 | 0 ~ 65,535 = 2¹⁶ − 1 |
| double word / dword (32-bit) | 4 | 32 | 0 ~ 4,294,967,295 = 2³² − 1 (약 42억) |
| quad word / qword (64-bit) | 8 | 64 | 0 ~ 2⁶⁴ − 1 (약 1800경) |

MBR 문서에 "16-bit word AA55" 라고 나오면 "2 byte 덩어리 AA55" 를 뜻한다. MBR 전체 크기 **512 bytes = 4,096 bits** 이다.

## §2. 16진수 (hexadecimal) — 0x 가 뭔데?

### 2-1. 2진수는 너무 길어서 못 읽는다

1 byte 를 2진수로 쓰면 8자리다: `11111111`. MBR 은 512 byte 니까 2진수로는 4,096자리. **사람이 못 읽는다.**

그래서 더 짧게 적을 방법이 필요한데, 10진수(0~9)는 컴퓨터의 2진수와 변환이 지저분하다 (2의 거듭제곱이 아님). 그래서 **2진수 4자리를 묶어서 한 글자로** 표현하는 방식이 나왔다 — **16진수**.

### 2-2. 16진수의 16개 자릿수

| 10진수 | 16진수 | 2진수 (4 bits) |
|---|---|---|
| 0~9 | 0~9 | 0000~1001 |
| 10 | **A** | 1010 |
| 11 | **B** | 1011 |
| 12 | **C** | 1100 |
| 13 | **D** | 1101 |
| 14 | **E** | 1110 |
| 15 | **F** | 1111 |

0~9 까지는 10진수와 같고, 10~15 는 **A, B, C, D, E, F** 로 쓴다.

### 2-3. 16진수 한 자리 = 2진수 4자리 = nibble

16진수 한 글자로 **4 bits** 를 표현한다. 이 4 bits 단위를 **nibble[^nibble]** 이라 한다. byte 는 8 bits 이므로 **16진수 2자리 = 1 byte**.

[^nibble]: **nibble** — "조금씩 갉아먹다" 라는 영어 단어. byte 의 절반이라 붙인 장난 용어. 공식 문서에도 쓰인다.

### 2-4. `0x` 접두어의 의미

숫자 `55` 를 보면 "이게 10진수 55 인지, 16진수 0x55(=10진수 85) 인지" 알 수가 없다. 그래서 **"뒤에 오는 숫자는 16진수다"** 라고 표시하는 접두어가 `0x` 다.

- `55` = 10진수 55
- `0x55` = 16진수 55 = 10진수 **85**
- `0xAA` = 16진수 AA = 10진수 **170**
- `0xFF` = 16진수 FF = 10진수 **255** (1 byte 최댓값)

### 2-5. 16진수 ↔ 10진수 계산법

16진수는 "16의 거듭제곱" 으로 자릿값이 매겨진다.

`0xAB12` 를 10진수로 바꾸는 법:

```
0xAB12
  ││││
  │││└─ 2 × 16⁰ = 2 × 1    = 2
  ││└── 1 × 16¹ = 1 × 16   = 16
  │└─── B × 16² = 11 × 256 = 2,816
  └──── A × 16³ = 10 × 4096 = 40,960
                       합계 = 43,794
```

MBR 에 자주 나오는 주소값:

| 16진수 | 10진수 | 의미 |
|---|---|---|
| `0x01BE` | 446 | 파티션 테이블 시작 위치 (MBR 안에서) |
| `0x01FE` | 510 | 부트 서명(55 AA) 시작 위치 |
| `0x7C00` | 31,744 | BIOS 가 MBR 을 읽어들이는 메모리 주소 |

### 2-6. 왜 2진수 대신 16진수를 쓰는가 — 실용성

같은 byte 를 세 방식으로 써보면:

| 표기법 | 글자 수 | 예시 |
|---|---|---|
| 2진수 | 8글자 | `10101010` |
| 10진수 | 최대 3글자 | `170` |
| 16진수 | **2글자** | `AA` |

16진수는 "짧고 + 2진수와 직결" 되는 장점이 있어서 **저수준 시스템 프로그래밍의 공용어**가 되었다.

## §3. endian — 바이트 순서 문제

### 3-1. 여러 byte 짜리 숫자를 메모리에 어떻게 놓는가

숫자 `0x12345678` (4 bytes)을 메모리에 저장한다고 치자. 메모리는 한 칸에 1 byte 씩 들어가는 선반이다. `12`, `34`, `56`, `78` 이 네 조각을 어느 순서로 넣을까?

두 가지 방식이 있다:

```
선반 주소:    100   101   102   103
             ┌───┬───┬───┬───┐

빅 엔디언:   │ 12│ 34│ 56│ 78│   ← 큰 자릿수가 앞
             └───┴───┴───┴───┘

리틀 엔디언: │ 78│ 56│ 34│ 12│   ← 작은 자릿수가 앞
             └───┴───┴───┴───┘
```

### 3-2. 이름의 유래

**endian** 은 조너선 스위프트의 소설 『걸리버 여행기』에 나오는 용어 "Lilliput 왕국에서 삶은 달걀을 넓은 쪽(big end) 부터 깨느냐, 좁은 쪽(little end) 부터 깨느냐로 전쟁한다" 에서 따온 말이다[^endian-origin]. 컴퓨터 공학자 Danny Cohen 이 1980년 논문에서 빌려왔다.

- **big-endian**: 큰 자릿수(big end) 를 먼저 저장
- **little-endian**: 작은 자릿수(little end) 를 먼저 저장

[^endian-origin]: Cohen, Danny (1980). *"On Holy Wars and a Plea for Peace"*. IETF IEN 137.

### 3-3. x86 은 little-endian

Intel x86 계열 CPU (우리 Born2beRoot VM 포함) 는 **little-endian** 이다. 따라서 메모리 덤프를 보면:

```
숫자:   0xAA55 (= 0xAA × 256 + 0x55 = 43,605)
메모리: 55 AA   (little-endian 이라 작은 자릿수 0x55 부터)
```

### 3-4. 서문의 한 문장이 드디어 읽힌다

Wikipedia 원문 재방문:

> *"The MBR signature will appear in a disk editor as the sequence **55 AA**. On x86 compatible machines, this can be written as 16-bit word **AA55** in programs for x86 processors (note the swapped order), whereas it would have to be written as **55AA** in programs for other CPU architectures using a big-endian representation."*

이제 해독 가능하다:

- **MBR 서명**: 숫자 `0xAA55` (16-bit word, 값 43,605)
- **disk editor 로 보면**: byte 순서대로 `55 AA` (x86 = little-endian)
- **x86 프로그램 소스 코드에서는**: `0xAA55` 라고 적음 (프로그램은 숫자 개념으로 다루므로)
- **big-endian 기계 프로그램에서는**: 메모리에 `55 AA` 로 저장되어야 하므로 소스 코드에서도 `0x55AA` 로 적어야 함

**한 마디로**: "디스크를 열어보면 `55 AA` 로 보이지만, 실제 숫자값은 `0xAA55` 다. little-endian 이라 바이트 순서가 뒤집혀 저장되기 때문."

### 3-5. MBR 에서 endian 이 왜 중요한가

파티션 테이블의 모든 여러 byte 필드 (파티션 시작 LBA[^LBA-pre], 섹터 개수 등) 는 전부 little-endian 으로 저장된다. 직접 디스크 섹터를 16진수로 읽을 때 byte 순서를 뒤집어야 진짜 숫자값을 얻는다.

[^LBA-pre]: **LBA** (Logical Block Addressing) — 섹터에 단순 번호(0, 1, 2, ...)를 매겨서 주소 지정하는 방식. §5 에서 자세히 설명.

**예시**: 파티션 엔트리의 LBA 필드가 디스크에서 `00 08 00 00` 으로 읽힌다면:
- little-endian 해석 → `0x00000800` = 10진수 2048
- 즉 "이 파티션은 디스크의 2048번 섹터부터 시작한다"

### 3-6. 자주 하는 혼동 — "16-bit" ≠ "16 bytes"

MBR 문서에서 "16-bit word" 라는 표현을 "16 바이트짜리 덩어리" 로 읽으면 모든 계산이 틀어진다. 반드시 구분:

- **16-bit = 2 bytes** (8 bits × 2)
- **16 bytes = 128 bits** (완전히 다른 크기!)

| 표현 | bit 수 | byte 수 | hex 자릿수 | 예시 값 |
|---|---|---|---|---|
| bit | 1 | 1/8 | — | 0 또는 1 |
| byte | 8 | 1 | 2자리 | `0xAA` |
| **word (16-bit)** | 16 | **2** | 4자리 | `0xAA55` |
| dword (32-bit) | 32 | 4 | 8자리 | `0xDEADBEEF` |
| qword (64-bit) | 64 | 8 | 16자리 | `0x1122334455667788` |

MBR 서명 `0xAA55` 는 **16-bit word = 2 bytes** 짜리 덩어리. "16 bytes" 덩어리(= 128 bits) 가 아니다. 16 bytes 는 파티션 엔트리 하나의 크기와 같다 (= 128 bits).

## §4. sector — 디스크가 읽고 쓰는 최소 단위

### 4-1. 하드디스크의 물리 구조 간단 버전

하드디스크(HDD) 는 원판(플래터) 이 회전하고, 그 위를 헤드가 떠다니며 자기로 데이터를 읽고 쓴다.

```
     플래터 (위에서 본 단면)
    ┌─────────────────┐
    │    ○ ○ ○ ○ ○    │
    │  ○           ○  │     ◊ 트랙 (track)
    │ ○   ┌─────┐   ○ │     = 동심원 고리
    │ ○   │     │   ○ │
    │ ○   │  ◈ │   ○ │     ◈ 섹터 (sector)
    │ ○   │     │   ○ │     = 트랙을 쪼갠 조각
    │ ○   └─────┘   ○ │
    │  ○           ○  │
    │    ○ ○ ○ ○ ○    │
    └─────────────────┘
```

### 4-2. 왜 최소 단위가 존재하는가

플래터가 1초에 수천 번 회전하는데, 헤드가 특정 byte 하나만 골라 읽는 건 물리적으로 불가능하다. 헤드가 지나가는 **고정된 덩어리** 를 통째로 읽고, 그 중 필요한 byte 만 뽑아 쓰는 방식이 유일하게 가능하다. 이 덩어리가 **섹터(sector)** 다.

### 4-3. 물리 섹터 vs 논리 섹터 — 지금부터 중요

섹터 개념에는 두 층이 있다. 이 둘이 같을 수도 있고 다를 수도 있다.

**물리 섹터 (physical sector)**
- 디스크 하드웨어가 실제로 **플래터에 자기로 기록하는** 영역의 크기
- 옛날 HDD: 512 bytes
- 현대 HDD/SSD: **4096 bytes (= 4 KB)** — Advanced Format (AF) 이라 부름
- 컨트롤러 펌웨어가 이 단위로 디스크 매체에 접근

**논리 섹터 (logical sector)**
- 디스크 컨트롤러가 **OS 에게 "이게 섹터 크기야" 라고 보고** 하는 크기
- 하위 호환성을 위해 대부분의 현대 디스크가 **512 bytes 로 보고** (실제 물리는 4096 bytes 여도)
- 이런 디스크를 **"512e"** (512-byte emulation) 라고 부른다
- OS (Linux, Windows) 가 `read()` 시스템 콜로 접근할 때 보는 섹터 크기

```
응용 프로그램 / OS 입장
    ↓ "512 byte 섹터로 보임"
디스크 컨트롤러 펌웨어
    ↓ (내부 변환)
    ↓ "실제는 4096 byte 물리 섹터"
플래터 (물리 매체)
```

### 4-4. 그래서 "sector 63", "sector 2048" 은 어느 기준?

**답: 대부분 논리 섹터 = 512 bytes 기준이다.**

이유: MBR 파티션 테이블에 기록된 "파티션 시작 LBA" 필드는 **논리 섹터 번호** 다. OS·파티션 도구·MBR 자체가 논리 섹터 시각으로만 세상을 본다. 물리 섹터 크기가 4096 bytes 든 512 bytes 든 무관하게 LBA 번호 체계는 논리 섹터 512 bytes 기준으로 돌아간다 (단, 네이티브 4K 디스크는 예외 — 거의 안 쓰임).

**계산 예시**:

- `sector 63` 부터 시작하는 파티션 → 디스크 앞에서부터 `63 × 512 = 32,256` bytes 지점 (약 31.5 KB)
- `sector 2048` 부터 시작하는 파티션 → 디스크 앞에서부터 `2,048 × 512 = 1,048,576` bytes 지점 (정확히 1 MB)

### 4-5. MBR 과 섹터의 관계

MBR 은 **디스크의 논리 섹터 0번 전체 512 bytes** 를 차지한다. 즉:

```
sector 0 (= 디스크 선두 0 ~ 511 byte) = MBR
    └─ 안에 446 bytes bootstrap + 64 bytes partition table + 2 bytes signature

sector 1 ~ (첫 파티션 시작 직전 섹터) = MBR 과 첫 파티션 사이의 "빈 공간"
    └─ 전통적으로는 sector 63 까지 (약 31 KB 여유)
    └─ 현대 표준은 sector 2048 까지 (약 1 MB 여유)

sector 63 또는 sector 2048 부터 = 첫 파티션(sda1) 시작
```

**MBR 과 첫 파티션 사이에 왜 빈 공간이 있나**: 역사적으로 CHS 주소 체계의 "트랙 1, 헤드 0, 섹터 1 부터 파티션 시작" 관습이 `LBA 63` (= 논리 섹터 63번) 에 해당했다. 현대에는 4 KB 물리 섹터와 정렬을 맞추기 위해 `LBA 2048` (= 1 MB 경계) 에서 시작한다.

**이 빈 공간의 용도는 제9장 (chain loading) 에서 설명한다**. 지금은 "빈 공간이 있다" 만 인지하면 된다.

### 4-6. "block" 과 "sector" 의 차이 — 명확화

여기서 **물리적/논리적 용어 혼선**이 가장 자주 일어난다. 바로 잡겠다.

| 용어 | 정의 | "물리/논리" 축에서의 위치 |
|---|---|---|
| **physical sector** (물리 섹터) | 디스크 플래터에 실제로 쓰이는 단위 (4 KB AF 또는 512 B) | **가장 물리적** |
| **logical sector** (논리 섹터) | 컨트롤러가 OS 에게 보고하는 단위 (보통 512 B) | 중간 |
| **block** (블록) | 파일시스템이 관리하는 단위 (예: ext4 기본 4 KB) | **가장 논리적** (위로 올라갈수록 추상) |

**축 정리**:

```
가장 아래                                              가장 위
(하드웨어에 가깝다)                                (추상화가 많다)
─────────────────────────────────────────────────────────────→

plate   →   physical sector   →   logical sector   →   block
(플래터)    (4 KB 자기 기록)      (512 B 환산 보고)      (FS 관리 단위)
            ↑ 당신이 느낀 "물리적"       ↑ 디스크 드라이버
            감각이 여기 해당          ↑ OS 가 보는 세계
                                            ↑ ext4 같은 파일시스템
```

**당신이 "block 이 더 물리적 느낌" 이라고 한 이유**는 "block" 이 영어로 "덩어리" 같은 실물 느낌이 있어서다. 하지만 **컴퓨터 과학 용어로는 block 이 더 위(추상) 계층** 이다.

`sector` = **디스크 하드웨어 수준** 이라는 표현의 정확한 의미:
- "하드웨어 수준" = **하드웨어(디스크) 가 정의한 단위다** 라는 뜻
- "논리적" 이라는 형용사는 "컴퓨터가 개념으로 다룬다" 는 뜻이지 "물리가 아니다" 는 뜻이 아님
- 논리 섹터도 여전히 **디스크 컨트롤러가 정의** 하므로 "하드웨어 수준"
- block 은 **파일시스템 소프트웨어가 정의** 하므로 "소프트웨어 수준"

**MBR 문서의 "first block" 은**: 하드웨어 문맥이므로 "첫 논리 섹터" 의 의미로 쓰인 것. 파일시스템 블록이 아님. Wikipedia 가 용어를 느슨하게 쓴 경우.

## §5. LBA — 섹터에 번호 매기는 방법

### 5-1. 섹터 주소는 어떻게 지정하나

디스크에 섹터가 수십억 개 있는데, "몇 번째 섹터 읽어줘" 라고 요청하려면 섹터마다 번호가 있어야 한다. 이 번호 매기는 방식이 두 가지 있다.

### 5-2. 옛 방식: CHS (Cylinder-Head-Sector)

1980년대 HDD 는 기계적 구조 그대로 주소를 매겼다.

```
CHS 주소 = (실린더 번호, 헤드 번호, 섹터 번호)

실린더 (cylinder) = 여러 플래터에서 같은 위치의 트랙 묶음
헤드 (head)       = 몇 번째 플래터의 어느 면을 읽는가
섹터 (sector)     = 트랙 안에서 몇 번째 조각인가
```

예: `CHS (100, 2, 15)` = "100번째 실린더의 3번째 헤드의 15번째 섹터"

**문제점**:
- CHS 숫자 3개를 특정 bit 수에 우겨넣다 보니 표현 한계가 생김
- 최대 주소 가능 용량 = 약 8 GB (24-bit CHS 의 한계)
- 현대 SSD 는 물리적으로 실린더·헤드가 없는데 CHS 흉내만 냄 → 가짜 값

### 5-3. 현대 방식: LBA (Logical Block Addressing)

**LBA** = 섹터에 **단순히 0, 1, 2, 3, ... 순서대로 번호만 매기는 방식**. 실린더·헤드 개념 버림.

```
LBA 0 = 디스크 맨 앞 섹터 (= MBR 자리)
LBA 1 = 두 번째 섹터
LBA 2 = 세 번째 섹터
...
LBA N = (N+1)번째 섹터
```

**물리 위치 계산**: `LBA × 섹터 크기 = 디스크 시작부터의 byte 오프셋`

예:
- LBA 63 × 512 bytes = 32,256 bytes (디스크 앞에서 31.5 KB 지점)
- LBA 2048 × 512 bytes = 1,048,576 bytes (정확히 1 MB 지점)

### 5-4. LBA 주소 공간의 크기

MBR 파티션 테이블의 LBA 필드는 **32-bit**. 따라서 표현 가능한 섹터 번호 범위:

$$0 \le \text{LBA} \le 2^{32} - 1 = 4,294,967,295$$

논리 섹터 크기 512 bytes 일 때 최대 디스크 크기:

$$2^{32} \times 512 \text{ bytes} = 2,199,023,255,552 \text{ bytes} = 2 \text{ TiB}$$

**이게 MBR 의 2 TB 한계의 수학적 이유다.** 2 TB 이상 디스크는 GPT 파티션 테이블을 써야 함.

### 5-5. CHS 와 LBA 공존

MBR 파티션 엔트리는 **CHS 와 LBA 를 둘 다 저장** 한다 (16-byte 엔트리 중 CHS 가 6 bytes, LBA 가 8 bytes). 하위 호환성 때문. 현대 OS 는 LBA 만 사용, CHS 필드는 무시 또는 형식적 값.

### 5-6. Born2beRoot 의 partman 화면에서 LBA 보는 법

설치기 파티션 화면의 각 파티션은 내부적으로 LBA 번호를 가진다. `cat /proc/partitions` 에서:

```
8    0  20971520  sda     ← 전체 디스크 20971520 블록 (1 block = 1024 bytes)
                            = 20971520 × 1024 = 약 21 GB
```

주의: `/proc/partitions` 의 `#blocks` 는 **1024-byte 블록 단위** 다 (kernel 의 특이 규칙). 섹터 개수가 아니다. 섹터는 보통 512 bytes 이므로 `#blocks × 2 = 섹터 개수 = LBA 개수`.

### 5-7. 경고 — LBA 와 segment:offset 은 완전히 다른 이야기다

**이 둘은 절대 섞어서 생각하지 말 것.** 제0장에서 가장 자주 실수하는 지점.

| 구분 | LBA | segment:offset |
|---|---|---|
| **용도** | **디스크** 주소 체계 | **메모리(RAM)** 주소 체계 |
| 어디에 저장? | MBR 파티션 엔트리 안 | CPU 레지스터 (CS/DS/ES + IP/SI/DI) |
| bit 수 | **32-bit** (MBR 기준) | **20-bit** (8086 기준) |
| 최대 주소 가능 범위 | 2³² × 512 bytes = **2 TiB** | 2²⁰ bytes = **1 MB** |
| 관련 한계 | "MBR 2 TB 디스크 한계" | "DOS 640 KB 메모리 한계" |

**MBR 의 2 TB 한계 수학적 증명**:

$$
\text{최대 디스크 크기} = (2^{32} - 1) \times 512 \text{ bytes} 
\approx 2^{32} \times 2^{9} = 2^{41} \text{ bytes} = 2 \text{ TiB}
$$

- MBR 파티션 엔트리의 LBA 필드 = **32-bit** 정수 (4 bytes)
- 최대 LBA 번호 = 2³² − 1 ≈ **43억**
- 1 LBA 당 512 bytes (논리 섹터)
- 총합 = 2⁴¹ bytes = **2 TiB**

**"64 KB 메모리 한계" 나 "16-bit 레지스터" 같은 숫자들은 여기서 나오지 않는다.** 그건 메모리 이야기. 2 TB 는 순수 **디스크 주소 폭** 이야기.

### 5-8. 왜 LBA 필드가 32-bit 인가 — 64-bit 로 만들면 안 됐나?

자연스러운 질문: "필드 크기를 64-bit 로 늘리면 9.4 ZB (제타바이트) 까지 지원 가능한데 왜 32-bit 로 고정했나? 혁명적으로 늘릴 수 있었을 텐데?"

**답: 1983년의 설계 제약 + 하위 호환성의 감옥.**

**1. 1983년 하드웨어 제약**:

MBR 포맷은 **IBM PC DOS 2.0 (1983)** 에 도입됐다. 당시 컴퓨터 환경:

| 항목 | 1983년 현실 |
|---|---|
| CPU | Intel 8086 = **16-bit** 레지스터 (32-bit 는 1985년 80386 에서야 등장) |
| 대중적 하드디스크 크기 | **10 MB** (XT 모델) — 당시 "엄청난 용량" |
| 32-bit 정수 | 소프트웨어가 다룰 수 있는 **현실적 최대치** (64-bit 는 에뮬레이션 필요, 매우 느림) |
| 2 TB | 10 MB 의 **21 만 배** — "평생 쓸 일 없는 무한" 이라는 인식 |

설계자의 전제: **"32-bit 면 평생 충분하다"**. 결과적으로 2010년대에 2 TB 디스크가 대중화되면서 한계에 부딪힘.

2^32 * 512 = 2^31 * 1024 = 2^31 * 2^10(KB) = 2^21 * 2^20(MB) = 2^11 * 2^30(GB) = 2^1 * 2^40(TB) 따라서 2TB

**2. 왜 지금 MBR 을 64-bit 로 업데이트하지 않나**:

- MBR 포맷은 **44 년간 수십억 대의 PC 에 고정된 표준**. 각 PC 의 BIOS, Windows, Linux 부트로더, 파티션 도구, 포렌식 도구가 **전부 32-bit LBA 를 가정** 하고 만들어짐.
- LBA 필드 크기 하나만 바꿔도 **모든 기존 도구가 작동 불능**. 
- 호환성을 유지하면서 한 bit 이라도 추가하는 건 불가능.

**3. 해결 — 새 포맷으로 교체 (MBR → GPT)**:

업데이트 대신 **완전히 새로운 이름, 새로운 포맷** 을 만들었다.

- **GPT (GUID Partition Table)**: UEFI 스펙에 포함된 2005년 도입
- **LBA 필드 = 64-bit** → 최대 주소 가능 = 2⁶⁴ × 512 = **9.4 ZB (제타바이트)** (현실적 무한)
- 파티션 최대 128 개 (MBR 의 4개에서 대폭 증가)
- CRC32 체크섬으로 손상 감지
- 2010년대 이후 **3 TB 이상 디스크는 전부 GPT** 로 포맷됨

**4. Born2beRoot 가 MBR 을 쓰는 이유**:

- VM 디스크 크기 ~21 GB → 2 TB 한계 안. 문제 없음.
- MBR 은 **구조가 단순** (512 bytes 한 섹터) → 학습 목적에 적합
- 평가자가 `lsblk`, `fdisk -l` 로 구조를 쉽게 검증 가능
- Debian 설치기 기본값 (BIOS 부팅 모드) 이라 자동 선택됨

**요약 — "왜 혁명으로 늘리지 않나"**:
- 1983 년 당시에는 32-bit 가 **현실적 최대**
- 지금은 **호환성 감옥** 때문에 기존 MBR 필드를 못 건드림
- 대신 **GPT 라는 새 포맷** 이 이미 혁명을 달성함 (64-bit LBA, 제타바이트급)
- MBR 은 **레거시 시스템용** 으로 남음

## §6. 메모리 주소와 segment:offset

### 6-1. 메모리는 거대한 byte 배열이다

RAM 은 **번호가 매겨진 1-byte 선반이 수십억 개** 늘어선 구조다. 각 선반의 번호가 **메모리 주소(address)** 다.

```
주소:  0x00000000   0x00000001   0x00000002   ...   0xFFFFFFFF
      ┌─────────┐ ┌─────────┐ ┌─────────┐     ┌─────────┐
      │  1 byte │ │  1 byte │ │  1 byte │ ... │  1 byte │
      └─────────┘ └─────────┘ └─────────┘     └─────────┘
```

32-bit 주소 공간이면 `0x00000000 ~ 0xFFFFFFFF` = 2³² ≈ 43억 개 선반 = **4 GB**.

### 6-2. 16-bit CPU 의 한계

Intel 8086 (1978년 출시, IBM PC 의 원조 CPU) 은 **레지스터가 전부 16-bit** 였다.

레지스터 하나로 표현 가능한 주소 = 2¹⁶ = **65,536 bytes = 64 KB**.

그런데 8086 의 **주소 버스는 20-bit** 였다. 즉 물리적으로는 최대 2²⁰ = **1,048,576 bytes = 1 MB** 의 메모리를 달 수 있었다.

**딜레마**: 20-bit 주소를 만들어야 하는데 레지스터는 16-bit 밖에 없음.

해법: **레지스터 두 개를 조합해서 20-bit 주소 생성**. 이게 **segment:offset** 방식이다.

### 6-3. segment:offset 표기법 읽는 법

`segment:offset` 표기는 **콜론(`:`) 을 기준으로 앞은 segment 값, 뒤는 offset 값**. 둘 다 **16-bit 숫자** (= 16진수 4자리) 다.

```
0x0000:0x7C00
│       │
│       └─ offset  = 0x7C00 (16-bit)
└───────── segment = 0x0000 (16-bit)
```

표기 규칙 예시:

| 표기 | segment | offset | 설명 |
|---|---|---|---|
| `0x0000:0x7C00` | 0x0000 | 0x7C00 | MBR 로드 주소의 표준 표기 |
| `0x07C0:0x0000` | 0x07C0 | 0x0000 | 같은 물리 위치의 다른 표기 |
| `CS:IP` | CS 레지스터 값 | IP 레지스터 값 | 실행 위치를 레지스터 이름으로 |
| `0040:0013` | 0x0040 | 0x0013 | 0x 접두어 생략된 형태 (문서마다 다름) |

### 6-4. 물리 주소 계산 공식의 수학적 유도

**질문: segment 와 offset 을 어떻게 합쳐 20-bit 주소를 만드나?**

목표:
- segment: 16-bit (0x0000 ~ 0xFFFF)
- offset: 16-bit (0x0000 ~ 0xFFFF)
- 합쳐진 물리 주소: 20-bit (0x00000 ~ 0xFFFFF)

16 + 16 = 32 bit 로 20-bit 주소를 만들려면 **겹치는 부분이 12 bit** 있어야 한다 (32 - 20 = 12).

Intel 의 설계:

$$\text{물리 주소} = \text{segment} \times 16 + \text{offset}$$

**"× 16" 의 수학적 의미**: 2진수에서 **왼쪽으로 4 bit shift** 하는 것. 왜냐면 16 = 2⁴.

```
segment (16-bit):         ssss ssss ssss ssss
segment × 16 (20-bit):    ssss ssss ssss ssss 0000  (왼쪽으로 4-bit shift, 오른쪽 4-bit에 0 채움)

offset (16-bit):              oooo oooo oooo oooo   (정렬을 맞춰서 더하기)

합산 (최대 21-bit*):      20-bit 물리 주소 생성
```

\* 실제로는 오버플로우까지 합쳐서 21-bit 가 될 수도 있으나 8086은 20-bit 주소 버스만 가지므로 최상위 bit 은 잘림 (주소 wrap-around).

### 6-5. 수학적 증명 — 예제로 확인

**예제 1**: `segment = 0x0000`, `offset = 0x7C00`

```
segment × 16 = 0x0000 × 0x10 = 0x00000
offset       =                 0x07C00
              ─────────────────────────
물리 주소    = 0x00000 + 0x07C00 = 0x07C00 = 31,744 (10진수)
```

**예제 2**: `segment = 0x07C0`, `offset = 0x0000`

```
segment × 16 = 0x07C0 × 0x10 = 0x07C00
offset       =                 0x00000
              ─────────────────────────
물리 주소    = 0x07C00 + 0x00000 = 0x07C00 = 31,744 (10진수)
```

**두 표기가 같은 물리 주소**. 이게 "여러 segment:offset 조합이 같은 물리 주소를 가리킨다" 의 정체.

**예제 3**: `segment = 0x1234`, `offset = 0x5678`

```
segment × 16 = 0x1234 × 0x10 = 0x12340
offset       =                 0x05678
              ─────────────────────────
물리 주소    = 0x12340 + 0x05678 = 0x179B8 = 96,696 (10진수)
```

### 6-6. 왜 같은 주소를 여러 방식으로 표기할 수 있나

같은 물리 주소를 표현하는 (segment, offset) 쌍이 **여러 개** 존재한다. 이유: segment 와 offset 의 표현 범위가 겹치기 때문.

물리 주소 `0x07C00` 을 가리키는 방법:

| segment | offset | segment × 16 + offset |
|---|---|---|
| 0x0000 | 0x7C00 | 0x00000 + 0x07C00 = 0x07C00 ✓ |
| 0x0100 | 0x6C00 | 0x01000 + 0x06C00 = 0x07C00 ✓ |
| 0x0200 | 0x5C00 | 0x02000 + 0x05C00 = 0x07C00 ✓ |
| 0x07C0 | 0x0000 | 0x07C00 + 0x00000 = 0x07C00 ✓ |

전부 같은 메모리 칸을 가리킨다. **표준 관행**: 16-bit boot code 의 경우 `segment=0`, `offset=전체 주소` 로 쓰는 게 일반적 (= `0x0000:0x7C00`).

### 6-7. `0x7C00` 의 수학적 증명 — 왜 하필 이 주소인가

MBR 이 로드되는 메모리 주소 `0x7C00` 의 역사적·수학적 이유.

**IBM PC (1981) 의 초기 메모리 맵 (real mode, 1 MB = 0x00000 ~ 0xFFFFF)**:

```
0x00000 - 0x003FF : IVT (Interrupt Vector Table, 1 KB)
0x00400 - 0x004FF : BDA (BIOS Data Area, 256 bytes)
0x00500 - 0x07BFF : 자유 메모리 (약 30 KB)
0x07C00 - 0x07DFF : ★ MBR 로드 위치 (512 bytes)
0x07E00 - 0x07FFF : MBR 스택 공간 (512 bytes) 
0x08000 - 0x9FFFF : 자유 메모리 (약 608 KB)
0xA0000 - 0xBFFFF : VGA 비디오 메모리
0xC0000 - 0xFFFFF : BIOS ROM 영역
```

**IBM PC 5150 (original) + 디스크 옵션의 최소 RAM 요구사항 = 32 KB**.

32 KB 를 16진수로 환산:
$$32 \text{ KB} = 32 \times 1024 = 32,768 \text{ bytes} = 0x8000$$

즉 IBM PC 5150 + 디스크의 최소 RAM 주소 공간 = `0x00000 ~ 0x07FFF` (32 KB).

**"32번째 KB 의 첫 byte"** 의 계산:
- 1번째 KB = `0x00000 ~ 0x003FF` (1024 bytes = 0x400 bytes)
- 2번째 KB = `0x00400 ~ 0x007FF`
- n번째 KB = `0x00000 + (n−1) × 0x400 ~ 0x00000 + n × 0x400 − 1`
- **32번째 KB 의 첫 byte** = `(32 − 1) × 0x400` = `31 × 0x400` = `0x7C00` ✓

**왜 32 KB 의 끝 부근에 MBR 을 놓나?**

설계 원칙: **"MBR 이 실행된 후에 더 큰 다음 단계 프로그램을 로드할 공간을 최대한 확보"**.

```
0x00000 ─┬── IVT + BDA (BIOS 관리 영역, 사용 불가)
         │
0x00500 ─┼── ┐
         │   │
         │   │  자유 공간: 약 30 KB
         │   │  → MBR 이 다음 단계 프로그램을 여기 로드
         │   │
0x07BFF ─┼── ┘
0x07C00 ─┼── ┐
         │   │  MBR 본체 (512 bytes)
0x07DFF ─┼── ┘
0x07E00 ─┼── ┐
         │   │  MBR 의 스택/작업 공간 (512 bytes)
0x07FFF ─┴── ┘  ← 32 KB 한계 (IBM PC 5150 + 디스크 최소 RAM)
```

**수학적 증명 완료**:
- 32 KB 한계 = `0x07FFF`
- MBR + 스택 여유 = 1 KB = `0x400`
- MBR 시작 = `0x08000 − 0x400` = `0x7C00` ✓

이 설계로:
- MBR 아래 **약 30 KB** 가 다음 단계 프로그램 로드용으로 확보됨
- MBR 위 **512 bytes** 가 MBR 의 스택/변수용으로 확보됨
- 최소 RAM 머신에서도 MBR 이 바깥으로 튀어나가지 않음

**이 값이 1981년에 결정된 이후 40년 넘게 안 바뀐다.** 하위 호환성의 화석.

### 6-8. 비트폭 참조표 — 절대 헷갈리지 말 것

시스템 문서에서 "XX-bit" 가 계속 나오는데 각자 **다른 의미** 다. 한 번에 정리:

| bit 수 | 16진수 자릿수 | 10진수 최댓값 | byte 수 | 통상 등장하는 곳 |
|---|---|---|---|---|
| 8-bit | 0x2자리 (0xFF) | 255 | 1 | byte 한 개, 개별 레지스터 일부 |
| **16-bit** | 0x4자리 (0xFFFF) | 65,535 | 2 | word, 8086 레지스터, offset 필드 |
| **20-bit** | **0x5자리** (0xFFFFF) | 1,048,575 | 2.5 | **8086 물리 주소 버스**, 1 MB 한계 |
| **32-bit** | 0x8자리 (0xFFFFFFFF) | 약 42억 | 4 | dword, x86 protected mode 주소, **MBR LBA 필드** |
| 64-bit | 0x16자리 | 약 1800경 | 8 | qword, x86_64 주소 |

**주의점 — 자가 답변에서 자주 나온 실수**:
- `0x5자리 ≠ 32-bit`. **0x5자리 = 20-bit** 가 맞다.
- `0x8자리 = 32-bit`. 이걸 `0x5자리` 로 착각하면 4자릿수 차이 (= 2¹² = 4096배) 나버림.
- segment:offset 의 결과값 `0xFFFFF` 는 **5자리 = 20-bit** 이지 32-bit 가 아니다.

**등장 비교**:
```
0x7C00    ← 16-bit 범위 안 (4자리). MBR 로드 주소. offset 으로 표현.
0x07C00   ← 20-bit 표기 (5자리). 같은 주소, segment:offset 물리 주소.
0x00007C00 ← 32-bit 표기 (8자리). 같은 주소, protected mode 에서 표시.
```

동일한 위치지만 해석하는 CPU 모드에 따라 자릿수가 달라진다. **5자리가 본질** 이고, 앞쪽 0 은 그냥 padding.

## §7. CPU register — CS, IP, DL 은 뭐냐

### 7-1. 레지스터란

CPU 내부에 있는 **초고속 저장소**. 메모리(RAM) 보다 수백~수천 배 빠르지만 개수가 매우 적다 (x86 에서 범용 16개 정도). CPU 가 계산하려는 값, 주소값, 상태 플래그 등을 여기 담는다.

비유: 메모리가 "거대한 도서관" 이라면 레지스터는 "책상 위에 펼쳐놓은 책 몇 권" 이다. 도서관에 가는 것보다 책상의 책을 보는 게 훨씬 빠르다.

### 7-2. x86 real mode 의 주요 레지스터들

| 레지스터 | 이름 | 용도 |
|---|---|---|
| **CS** | Code Segment | 실행 중인 코드의 segment |
| **IP** | Instruction Pointer | 다음 실행 명령의 offset |
| **DS** | Data Segment | 읽어올 데이터의 segment |
| **SI** | Source Index | 읽어올 데이터의 offset |
| **ES** | Extra Segment | 쓰기 대상 segment |
| **DI** | Destination Index | 쓰기 대상 offset |
| **SS** | Stack Segment | 스택의 segment |
| **SP** | Stack Pointer | 스택 top 의 offset |
| **AX, BX, CX, DX** | 범용 | 계산용 (DX 는 8-bit 절반 DL, DH 로 쪼개짐) |

`CS:IP` 같은 표기 = `CS 레지스터 값 : IP 레지스터 값` = `현재 실행 위치의 segment:offset`. §6-3 의 표기법과 동일.

### 7-3. BIOS 가 MBR 실행 전에 미리 설정하는 값

- **CS:IP** = `0x0000:0x7C00` (실행 시작 위치)
- **DL** = 부팅 드라이브 번호 (첫 HDD 면 `0x80`, CD-ROM 이면 `0x00` 등)
- **DS:SI** = 파티션 테이블 엔트리 위치 (선택 사항)

### 7-4. 왜 MBR 문서에 레지스터 이름이 줄줄 나오나

MBR 의 boot code 는 **어셈블리어로 쓴 446 바이트짜리 프로그램** 이다. 어셈블리는 CPU 레지스터에 직접 값을 쓰고 읽는 언어라, MBR boot code 가 "어느 레지스터에 뭘 넣고 JMP 하는지" 를 설명하려면 레지스터 이름이 필수다.

제6장 (BIOS→MBR→VBR 인터페이스) 에서 `DL = 0x80`, `CS:IP = 0x0000:0x7C00` 같은 표기가 대량 등장한다.

### 7-5. 왜 `CS:IP` 조합인가 — `CS:CP` 가 아닌 이유

직관적으로 보면 segment 레지스터가 모두 `-S` 로 끝나니까 (`CS`, `DS`, `ES`, `SS`), offset 쪽도 `CP`, `DP`, `EP` 같은 대응이 있을 것처럼 보인다. **하지만 Intel 은 그렇게 이름을 짓지 않았다.** offset 레지스터는 **"자기가 하는 일"** 을 기준으로 이름을 붙였다.

**Offset 레지스터의 이름 유래**:

| 이름 | 풀네임 | 의미 | 관습적 짝이 되는 segment |
|---|---|---|---|
| **IP** | **I**nstruction **P**ointer | 다음 실행할 명령의 위치 | **CS** |
| **SI** | **S**ource **I**ndex | 데이터 **소스(출발점)** 인덱스 | **DS** |
| **DI** | **D**estination **I**ndex | 데이터 **목적지(도착점)** 인덱스 | **ES** |
| **SP** | **S**tack **P**ointer | 스택 꼭대기 위치 | **SS** |
| **BP** | **B**ase **P**ointer | 스택 프레임 밑동 위치 | **SS** |

**통상 페어링 (조립 관습)**:
- `CS:IP` — "코드 세그먼트 : 명령 포인터" = **현재 실행 위치**
- `DS:SI` — "데이터 세그먼트 : 소스 인덱스" = **복사/읽기의 출발점**
- `ES:DI` — "엑스트라 세그먼트 : 목적지 인덱스" = **복사/쓰기의 도착점**
- `SS:SP` — "스택 세그먼트 : 스택 포인터" = **스택 top**

**핵심 원칙**: **세그먼트 레지스터 이름은 `-S` 로 통일**, **offset 쪽 이름은 기능으로** 정함. 페어링(CS:IP, DS:SI, ES:DI, SS:SP)은 **관습** 이지 이름 대칭이 아님.

**IP 의 역사적 유래**: Intel 8080 (1974) 이전 시대에는 이 레지스터를 **`PC` (Program Counter)** 라 불렀다. 1978년 8086 에서 segment 개념을 도입할 때 Intel 이 `PC` 를 `IP` (Instruction Pointer) 로 바꿨다. 그 결과 페어링이 **`CS:IP`** 가 됐다. 만약 `PC` 로 유지했으면 오늘날 `CS:PC` 였을 것. **역사적 우연**.

그래서 `CS:CP` 가 아니라 `CS:IP`. 당신의 직관적 대칭 기대가 틀린 게 아니라 Intel 이 대칭으로 짓지 않은 것이다.

### 7-6. IP 는 "현재 명령" 인가 "다음 명령" 인가 — 엄밀한 정의

자주 혼동되는 지점. 위 §7-5 표에서 `IP` 를 "다음 실행할 명령의 위치" 라고 썼는데, 다른 문서에서는 "현재 실행 중인 명령의 위치" 라고 쓰기도 한다. **어느 쪽이 맞나?**

**엄밀한 답: IP 는 "CPU 가 다음에 fetch 할 명령어의 offset"**.

**fetch-execute 사이클의 흐름**:

```
┌─ Cycle N ──────────────────────────────────────┐
│ 1. IP 가 가리키는 주소에서 명령어 A fetch       │
│ 2. IP ← IP + sizeof(A)                           │  ← A 실행되기 전에 IP 는 이미
│    (예: A 가 3 bytes 면 IP += 3)                 │     B 주소를 가리킴!
│ 3. 명령어 A execute                              │
│    (A 실행 중 IP 는 "다음 명령 B" 의 주소)      │
└──────────────────────────────────────────────────┘
┌─ Cycle N+1 ────────────────────────────────────┐
│ 1. IP (= B 주소) 에서 명령어 B fetch             │
│ 2. IP += sizeof(B)                               │
│ 3. B execute                                     │
└──────────────────────────────────────────────────┘
```

**핵심**:
- IP 는 "실행 중인 A" 의 주소가 아니라 **"다음에 fetch 할 B"** 의 주소.
- A 는 자기 자신의 주소를 **직접 알 수 없음** (별도 기법 필요, 예: `call $` 으로 스택에서 꺼내기).

**그럼 왜 "현재 실행 위치" 라고 표현하기도 하는가**:

교육·문서에서 느슨한 표현을 쓰는 이유:
- 명령 A 와 B 의 주소 차이는 평균 1–5 bytes (매우 근접)
- "실행 흐름이 이 동네 어딘가에 있다" 는 대략적 위치를 말하는 맥락
- 정확한 bit/byte 단위 차이는 어셈블리 디버깅 외엔 거의 안 중요

**정확한 표현 vs 느슨한 표현**:

| 맥락 | 표현 |
|---|---|
| 엄밀 (스펙, 어셈블리 매뉴얼) | "다음에 fetch 할 명령의 offset" |
| 느슨 (교육, 개요 설명) | "현재 실행 흐름의 위치" |
| 혼용할 때 | "다음 명령 = 현재 실행 흐름" 이라는 단서 붙임 |

**왜 여전히 중요한가 — JMP 명령 분석**:

`JMP 0x0000:0x7C00` 같은 명령의 효과:
- CS ← 0x0000
- IP ← 0x7C00
- **즉시 다음 fetch 부터 0x7C00 에서 명령을 읽어옴**

즉 JMP 는 "IP 를 특정 주소로 변경" 하는 명령. 당신이 느슨하게 "0x7C00 으로 점프" 라고 표현하는 것은 **"다음 fetch 가 0x7C00 에서 일어나게 만드는 것"** 과 정확히 같은 의미.

**당신의 지적이 맞다** — 앞선 §7-5 표에 "현재 실행 명령의 위치" 라고 쓴 건 **느슨한 표현**이었고, 엄밀히는 "다음에 fetch 할 명령의 위치" 가 정확. 교육적 맥락에서 혼용되지만 둘의 차이를 알고 있어야 JMP/CALL/RET 같은 제어 흐름 명령을 정확히 이해할 수 있다.

### 7-7. 실행 추적 — JMP 직후 CS:IP 가 어떻게 변하는가

`JMP 0x0000:0x7C00` 같은 명령을 처음 본 사람은 흔히 다음 같은 시나리오를 상상한다:

> "JMP 직후 CS = 0x7C00, IP = 0x8000 이 되었다고 치자. 그럼 다음 fetch 주소가 0x7C000 + 0x8000 = 0x84000 인가? 그런데 이거 쓰레기 주소 아닌가?"

이 한 줄에 **3 가지 오해** 가 섞여 있다. 하나씩 풀어보자.

#### 7-7-1. 오해 ① — `JMP 0x0000:0x7C00` 후 **CS 는 0x0000 이지 0x7C00 이 아니다**

`JMP segment:offset` 명령의 효과:

| 표기 | CS 가 받는 값 | IP 가 받는 값 |
|---|---|---|
| `JMP segment:offset` | **segment** (콜론 앞) | **offset** (콜론 뒤) |
| `JMP 0x0000:0x7C00` | **0x0000** | **0x7C00** |
| `JMP 0x07C0:0x0000` | **0x07C0** | **0x0000** |
| `JMP 0x1000:0x0010` | **0x1000** | **0x0010** |

**0x7C00 은 IP 가 받는 값이지 CS 가 받는 값이 아니다.** "0x7C00 으로 점프했으니 CS 도 0x7C00" 은 자연스러운 직관이지만 틀림.

#### 7-7-2. 오해 ② — CS 는 매 명령마다 바뀌지 않는다

가장 큰 혼동의 원인. **대부분의 명령 실행에서는 IP 만 자동 증가하고 CS 는 변하지 않는다.**

`JMP 0x0000:0x7C00` 직후 메모리에 MBR 코드가 있다고 가정 (예시):

```
주소        바이트         어셈블리
─────────────────────────────────────────────
0x07C00:   BB 00 80    ← MOV BX, 0x8000  (3 bytes)
0x07C03:   B9 00 02    ← MOV CX, 0x0200  (3 bytes)
0x07C06:   8E DB       ← MOV DS, BX      (2 bytes)
0x07C08:   ...
```

실행 추적:

| 시점 | CS | IP | 물리 주소 (CS×16+IP) | 일어나는 일 |
|---|---|---|---|---|
| JMP 직후 | **0x0000** | 0x7C00 | 0x07C00 | `MOV BX, 0x8000` fetch |
| 1번째 명령 실행 후 | **0x0000** | 0x7C03 | 0x07C03 | `MOV CX, 0x0200` fetch (IP 만 +3) |
| 2번째 실행 후 | **0x0000** | 0x7C06 | 0x07C06 | `MOV DS, BX` fetch (IP 만 +3) |
| 3번째 실행 후 | **0x0000** | 0x7C08 | 0x07C08 | 다음 명령 fetch (IP 만 +2) |
| ... | **0x0000** | 0x7CXX | 0x07CXX | CS 그대로, IP 만 증가 |

**CS 는 변하지 않는다.** IP 만 매 명령마다 그 명령의 길이만큼 자동 증가.

**CS 가 바뀌는 경우는 특별한 명령에서만**:

| 명령 | 효과 |
|---|---|
| **Far JMP** (segment 명시) | `JMP 0x1000:0x0000` → CS = 0x1000, IP = 0x0000 |
| **Far CALL** (segment 명시) | `CALL 0x1000:0x0000` → 스택에 옛 CS:IP 푸시 후 CS:IP 변경 |
| **Far RET** | 스택에서 CS:IP 둘 다 복원 |
| **INT** (인터럽트) | IDT (Interrupt Descriptor Table) 에서 새 CS:IP 가져옴 |
| 리셋 | CS:IP 초기값 (0xFFFF:0x0000) 으로 |
| 일반 명령 (MOV, ADD, JMP near 등) | **CS 변경 없음, IP 만 증가** |

즉 **CS 변경 = "다른 segment 의 코드로 건너뛴다" 는 특별한 행위**. 일반 코드 흐름에서는 거의 안 일어남.

#### 7-7-3. 오해 ③ — 0x84000 은 "쓰레기 주소" 가 아니다

산수 먼저: **0x7C000 + 0x8000 = ?**

```
   0x7C000   = 0111 1100 0000 0000 0000  (20-bit)
 + 0x08000   = 0000 1000 0000 0000 0000
 ─────────────────────────────────────────
   0x84000   = 1000 0100 0000 0000 0000  ← 정답
```

흔한 오답: `0x83000`. 5번째 자리에서 `C + 8 = 14 = 0x14` (carry 발생) 인데, 이걸 빼먹고 `B + 8 = 13 = 0xD` 처럼 처리하면 `0x83000` 이 나옴. 정답은 **0x84000** (= 10진수 540,672 = 약 528 KB).

이제 핵심: **0x84000 은 유효한 주소다.** Real mode 메모리 맵:

```
0x00000 - 0x003FF : IVT (Interrupt Vector Table) — 1 KB
0x00400 - 0x004FF : BDA (BIOS Data Area) — 256 bytes
0x00500 - 0x07BFF : 자유 conventional memory (~30 KB)
0x07C00 - 0x07DFF : MBR 위치 (BIOS가 여기에 로드)
0x07E00 - 0x9FFFF : 자유 conventional memory (~608 KB)  ← 0x84000 여기!
0xA0000 - 0xBFFFF : VGA video memory
0xC0000 - 0xEFFFF : 확장 ROM/BIOS
0xF0000 - 0xFFFFF : System BIOS ROM
```

**0x84000 은 자유 conventional memory 한가운데.** MBR 이 다음 단계 프로그램을 로드하기에 적합한 위치 중 하나.

**"쓰레기" 주소가 되는 경우** (구분 기준):

| 상황 | 결과 |
|---|---|
| `0xA0000` 이상 (RAM 이 아닌 video/ROM 영역) 으로 코드 점프 | 잘못된 동작, 종종 시스템 크래시 |
| `0x100000` 이상 (real mode 1 MB 한계 초과) | Wrap-around (HMA 예외 외엔 0 으로 돌아옴) |
| 코드가 아직 로드되지 않은 RAM 위치로 점프 | 무작위 바이트를 명령어로 해석 → 십중팔구 크래시 |
| **conventional memory (0x00500 ~ 0x9FFFF) 안의 정상 RAM** | **유효 — 정상 동작** |

당신이 가정한 0x84000 은 마지막 줄에 해당. **완벽히 유효한 주소.**

#### 7-7-4. 핵심 3 원칙 — 외워두면 CS:IP 헷갈릴 일 없음

1. **`JMP segment:offset` → CS = segment, IP = offset.** 콜론 앞이 CS, 콜론 뒤가 IP. 절대 헷갈리지 말 것.
2. **CS 는 매 명령마다 안 바뀐다.** IP 만 자동 증가. CS 는 far JMP/CALL/RET/INT 같은 특별 명령에서만 변경됨.
3. **`CS × 16 + IP` 의 모든 결과 (1 MB 안) 는 유효 주소.** "쓰레기" 여부는 메모리 맵상의 위치만 결정 (RAM 이냐 ROM 이냐 video 이냐).

#### 7-7-5. 왜 이 설계인가 — segment 의 의도

Intel 8086 이 segment:offset 을 도입한 이유 (지금 보면 의외):

- **64 KB 안의 코드 흐름은 효율적**: 한 segment 안에서 IP 만 증가시키며 실행. 매번 20-bit 주소 재계산 안 함.
- **"의미 있는 영역" 으로 메모리를 분할**: CS 는 코드, DS 는 데이터, SS 는 스택. 같은 offset (예: 0x100) 도 어느 segment 냐에 따라 다른 데이터.
- **재배치 (relocation) 용이**: 같은 코드를 다른 메모리 위치에서 실행하려면 segment 만 바꾸고 offset 은 그대로 유지.

오늘날 protected mode / long mode 에서는 segment 가 거의 무력화 (flat memory model) 되었지만, real mode 부팅 코드에는 여전히 살아 있음. MBR boot code 가 이 모델 안에서 동작.

## §8. real mode — 2026년 CPU 가 왜 1978년 모드로 깨어나는가

### 8-1. x86 의 3 가지 주요 동작 모드

| 모드 | 등장 | 주소 공간 | 특징 |
|---|---|---|---|
| **real mode** | 1978 (8086) | 1 MB (20-bit) | segment:offset, 단일 태스크, 메모리 보호 없음 |
| protected mode | 1982 (80286) | 4 GB (32-bit) | 가상 메모리, 특권 레벨, 멀티태스킹 |
| long mode | 2003 (AMD64) | 16 EB (64-bit) | 64-bit 레지스터, 현대 OS 의 기본 |

### 8-2. real mode 의 제약

- 메모리 총 1 MB 만 접근 가능 (그 중 640 KB 만 일반 코드용, 나머지는 BIOS/비디오 예약)
- segment:offset 로만 주소 계산
- 메모리 보호 없음 (아무 프로그램이나 다른 프로그램의 메모리 수정 가능)
- 멀티태스킹 불가
- **CPU 가 하나의 프로그램만 돌린다**

### 8-3. 그런데 왜 지금도 real mode 부팅?

**답: 40년 하위 호환성.**

IBM PC (1981) 가 Intel 8086 을 썼고, 8086 은 real mode 만 지원했다. 이후 CPU 가 발전해도 "PC 호환" 이려면 전원 켜자마자 real mode 로 깨어나야 한다는 약속을 Intel 이 지키고 있다. 2026년의 최신 Intel CPU 도 전원 ON 후 처음 몇 밀리초는 real mode 다.

### 8-4. 부팅 과정에서 모드 전환

```
전원 ON
   ↓
BIOS 실행 (real mode)                    ← 16-bit
   ↓
MBR boot code (real mode)                ← 16-bit
   ↓
GRUB stage 1 = boot.img (real mode)      ← 16-bit
   ↓
GRUB stage 1.5 = core.img                ← 부분적으로 protected mode 로 전환
   ↓
GRUB stage 2 = normal.mod (protected)    ← 32-bit
   ↓
Linux kernel 로드
   ↓
kernel 이 long mode 로 전환              ← 64-bit
   ↓
systemd, init 등
```

MBR 내용이 "real mode 기계어" 라는 말은 이 때문이다. MBR boot code 는 **16-bit 실행 환경** 에서 돌아간다.

## §9. bootstrapping — "부팅" 이라는 단어의 정체

### 9-1. 어원

영어에 `"pull oneself up by one's bootstraps"` 라는 관용구가 있다. 직역하면 "자기 신발끈을 당겨 스스로 일어선다". 논리적으로는 불가능한 일을 혼자 힘으로 해낸다는 뜻.

컴퓨터 전원 켜는 순간 메모리에는 아무것도 없다. OS 도 없다. 그런데 "OS 를 실행하려면 OS 코드를 메모리에 올려야 하고, 메모리에 올리려면 그걸 올리는 프로그램이 필요한데, 그 프로그램은 어디서 오냐?" 라는 닭과 달걀의 역설.

해답은 **여러 단계로 나눠서 한 단계씩 조금 더 큰 프로그램을 불러오는** 방식이다. 이 연쇄 과정이 "자기 힘으로 일어서는" 것 같다 해서 `bootstrapping`, 줄여서 `booting` 이라 부른다.

### 9-2. chain loading 의 필연성 — 각 단계의 존재 이유 (Q&A)

**Q1: BIOS 가 직접 OS 커널을 로드하면 안 되나?**

A: 안 된다. BIOS 는 1980년대 설계로 **파일시스템 개념이 없다**. 디스크의 **특정 섹터(sector 0) 하나** 만 읽을 수 있을 뿐이다. "`vmlinuz` 라는 이름의 파일을 찾아서 로드해라" 같은 요청은 못 한다. 파일 이름·파일시스템·디렉토리는 BIOS 의 인지 범위 밖.

**Q2: 그러면 왜 MBR 의 446 bytes 에 OS 를 넣지 못하나?**

A: 현대 OS 커널(Linux vmlinuz) 만 해도 **수 MB 단위**. 446 bytes 는 그 0.01% 도 안 된다. 파일시스템 드라이버 하나도 못 집어넣는 용량이다. MBR boot code 는 물리적으로 "다음 단계로 점프" 기능 외에는 수용 불가.

**Q3: 왜 GRUB 이 stage 1, 1.5, 2 세 단계로 나뉘나?**

A: **세 공간 각자의 물리적 제약이 다르기 때문**. 닭-달걀 문제를 끊는 데 세 단계가 필요하다.

GRUB 이 해야 할 세 가지 일을 상상해보자:

1. **MBR(첫 섹터 512 bytes)에서 시작한다** ← 이건 BIOS 의 제약
2. **파일시스템(ext4) 안에서 파일 이름으로 커널·config 를 찾아 읽는다** ← 복잡한 드라이버 필요
3. **사용자에게 메뉴 표시 + 키보드 입력 처리 + 커널 로드** ← 풍부한 기능

이 세 가지를 **한 프로그램**에 다 넣을 수 없는 이유:

**문제 1 — MBR 은 440 bytes 만 사용 가능.**
파일시스템 드라이버 하나(ext2 만 해도 수 KB)도 못 들어간다. 그래서 MBR 에 들어가는 **stage 1 (boot.img)** 은 **오직 "다음 섹터로 점프" 만** 할 수 있다.

**문제 2 — 닭-달걀 문제.**
파일시스템 드라이버는 어딘가에 **저장되어야** 한다. 파일시스템 **안에** 저장하면 "파일시스템을 읽으려면 드라이버가 필요한데 드라이버를 읽으려면 파일시스템이 필요" 모순. 해결: **파일시스템 바깥의 고정 섹터 위치** 에 저장한다. MBR 과 첫 파티션 사이의 **빈 공간 (약 1 MB)** 이 그곳이다. 이 공간은 파일시스템이 아니므로 stage 1 이 **섹터 번호로 직접 접근** 가능. 이것이 **stage 1.5 (core.img)** = 파일시스템 드라이버 내장한 중간 단계.

**문제 3 — stage 1.5 도 용량 제한.**
MBR-sda1 사이 빈 공간은 크지만 그래도 1 MB 정도. 현대 GRUB 의 풀 기능(메뉴 UI, 테마, LUKS 암호화 지원, 키보드 다국어, 네트워크 부팅 등)은 이 공간에 다 못 들어간다. 그래서 stage 1.5 는 "ext2 까지만 읽는 최소 기능" 만 갖고, 나머지는 `/boot/grub/` 에 있는 **stage 2 (normal.mod + 각종 .mod 파일)** 가 담당.

**닭-달걀을 세 단계로 끊는 구조**:

```
[MBR 440 bytes]                     [MBR-sda1 빈 공간 ~1 MB]              [파일시스템 안 /boot/grub/]
┌─────────────┐                    ┌──────────────────┐                  ┌──────────────────────┐
│  stage 1    │── "섹터번호 로드" ──→│  stage 1.5        │── "파일명 로드" ──→│  stage 2             │
│  boot.img   │                    │  core.img         │                  │  normal.mod + ...    │
│             │                    │  (ext2 드라이버)  │                  │  (풀 기능 부트로더)  │
│  "점프만"   │                    │                   │                  │                      │
└─────────────┘                    └──────────────────┘                  └──────────────────────┘
    고정 위치                         고정 섹터 위치                          파일시스템 안
    (BIOS 가 로드)                    (stage 1 이 섹터 번호로 찾음)           (stage 1.5 가 이름으로 찾음)
```

**비유 — 3단 로켓**:
- **stage 1** = 1단 부스터: 작은 폭발, 2단 분리만 함. 수명 몇 초.
- **stage 1.5** = 2단 부스터: 대기권 통과(= 파일시스템 인식). 임무 끝나면 버려짐.
- **stage 2** = 페이로드(위성): 실제 임무 수행 (사용자 메뉴, 커널 로드).

각 단계는 **자기 물리적 위치의 제약** 안에서 **다음 단계를 집어서 꺼내는** 임무만 한다. 한 단계로 다 하려고 하면 물리적으로 불가능.

**Q4: 왜 initramfs 가 필요하나?**

A: **커널 혼자서는 "복잡한 루트 파일시스템" 을 마운트할 방법이 없기 때문**. Born2beRoot 처럼 `LUKS → LVM → ext4` 로 겹친 루트가 특히 그렇다.

**단순한 경우 (initramfs 불필요)**:

루트가 `/dev/sda1` 처럼 단순 ext4 파티션 하나면:
- 커널 부팅 파라미터에 `root=/dev/sda1` 만 알려주면 됨
- 커널 내장 ext4 드라이버가 직접 마운트. 끝.

**Born2beRoot 의 경우 (initramfs 필수)**:

루트 `/` 까지 가는 길:

```
/dev/sda             ← 물리 디스크
  ↓
/dev/sda5            ← LUKS 암호화 컨테이너 (암호 풀어야 내용 보임)
  ↓ (cryptsetup luksOpen 필요)
/dev/mapper/sda5_crypt   ← 해독된 블록 디바이스 (이제야 LVM 이 보임)
  ↓ (lvm vgchange -ay 필요)
/dev/mapper/namykim42--vg-root   ← LVM 논리 볼륨 (진짜 / 가 여기)
  ↓ (mount 필요)
/                    ← 마침내 마운트!
```

**4 단계 작업** 이 필요한데, 각 단계에 **사용자 공간 실행 파일** 이 필요하다:

- `cryptsetup` (LUKS 해독, 암호 물어봄)
- `lvm` (볼륨 그룹 활성화)
- `mount` (ext4 마운트)

**문제**: 커널은 이런 실행 파일을 **내장하고 있지 않다**. 이 파일들은 전부 **파일시스템 안** 에 있는데, 그 파일시스템이 바로 우리가 마운트하려는 대상이다. **닭-달걀 문제 또 등장.**

**해결 — initramfs**

`initramfs` = **"init**ial **RAM f**ile **s**ystem" = **초기 램 파일시스템**.

구체적으로 `/boot/initrd.img-6.1.0-*` 같은 파일. 이게 뭐냐면:
- **cpio 아카이브를 gzip 으로 압축한 파일**
- 풀면 작은 루트 파일시스템 모양 (bin, sbin, etc, lib, init 등)
- 안에 **cryptsetup, lvm, mount 실행 파일** 이 들어있음
- 부팅 시 필요한 **커널 모듈 (dm-crypt, dm-mod, ext4 등)** 도 들어있음
- 최상위에 `/init` 이라는 셸 스크립트

**부팅 시 실제 일어나는 일**:

```
1. GRUB 이 메모리에 두 파일을 로드:
   - vmlinuz-*     (Linux 커널, ~20 MB)
   - initrd.img-*  (initramfs, ~50 MB 압축)

2. 커널 시작.

3. 커널이 initramfs 를 RAM 안에 풀고 "임시 / " 로 마운트.
   → 이 시점에 / 안에는 cryptsetup, lvm 등이 들어있음

4. 커널이 initramfs 의 /init 스크립트 실행.

5. /init 스크립트가 순서대로:
   ├─ dm-crypt 모듈 로드 (modprobe dm-crypt)
   ├─ "Please unlock disk sda5_crypt:" 메시지 출력
   ├─ cryptsetup luksOpen /dev/sda5 sda5_crypt  ← 사용자 암호 입력!
   ├─ dm-mod, lvm 모듈 로드
   ├─ vgchange -ay  ← LVM VG 활성화
   ├─ mount /dev/mapper/namykim42--vg-root /new_root  ← 진짜 / 마운트
   └─ exec switch_root /new_root /sbin/init  ← 임시 / 에서 진짜 / 로 전환

6. initramfs 는 메모리에서 사라짐 (해제).

7. systemd (= /sbin/init) 가 진짜 / 에서 시작.
```

**비유 — 이사 트럭**:

```
빈 집 (= 메모리)에 입주하려는데 가구(= OS 환경)가 없다.
  ↓
이사 트럭(initramfs)이 필요 도구(cryptsetup, lvm, mount)를 싣고 옴
  ↓
트럭 안에서 열쇠 작업(LUKS 해독) 하고 창고 열기(LVM 활성화)
  ↓
진짜 집(root LV)에 가구 배치(/ 마운트)
  ↓
이사 완료 → 트럭은 떠남(메모리에서 initramfs 해제)
  ↓
새 집에서 일상(systemd 서비스) 시작
```

**핵심 요약**: initramfs 는 **"진짜 루트를 마운트할 준비를 하는 임시 환경"** 이다. 단순 시스템은 필요 없지만 Born2beRoot 처럼 암호화·LVM 이 얽힌 시스템은 **필수**.

**이름의 구성**: `init` (initial) + `ram` + `fs` (file system) = "초기 램 파일시스템". 옛날엔 `initrd` 였고 지금은 `initramfs` 로 발전 (차이는 내부 구조, 기능은 같음).

**Q5: 왜 systemd 가 따로 필요하나?**

A: **커널은 스케줄러이고 사용자 공간이 아니다**.

- 커널의 역할: 메모리 관리, 프로세스 스케줄링, 시스템 콜 처리, 드라이버
- 사용자가 쓰는 것: bash, sshd, cron, ufw 같은 **프로세스들**
- 이 프로세스들을 **시작·정지·관리** 하는 주체가 필요 → `init` (전통) 또는 `systemd` (현대)
- systemd 가 `/etc/fstab` 읽어서 다른 파티션 (`/home`, `/var`, ...) 마운트, `ssh.service` 시작, 로그인 프롬프트 표시 등

### 9-3. 단계 수는 왜 늘어났는가 — 진화사적 관점

1980년대 IBM PC 초창기:

```
BIOS → MBR → VBR → OS (3~4 단계)
```

- OS 가 단순 (DOS: 파일시스템은 FAT12, 커널 수 KB, 단일 작업)
- MBR → VBR → OS 로 충분

요구사항 변화:

| 요구 | 추가된 단계 |
|---|---|
| 커널이 MB 단위로 커짐 | GRUB 같은 **전용 부트로더** 등장 |
| 다양한 파일시스템 지원 | GRUB 안에 파일시스템 드라이버 모듈 → **stage 1.5** 분리 |
| 디스크 전체 암호화 요구 | initramfs 의 **LUKS 해독 단계** 추가 |
| LVM 등 복잡한 볼륨 관리 | initramfs 의 **LVM 활성화 단계** 추가 |
| SysV init 의 병렬성·의존성 관리 한계 | **systemd** 등장 |
| EFI/UEFI 펌웨어 전환 | GRUB EFI 경로 / shim bootloader (Secure Boot) |

**원칙**: 한 단계가 모든 걸 해결하려 하면 그 단계가 지나치게 거대해진다. 각 단계에 "**자기 단계에서 해결 가능한 문제만**" 맡기고 나머지는 다음 단계에 넘기는 게 설계상 더 안전·호환적.

### 9-4. 하위 호환성이 단계 수를 보존하는 이유

**핵심 통찰**: 새로운 기능이 추가되어도 **오래된 단계는 절대 사라지지 않는다**. 40년 전의 IBM PC 호환 부팅 시퀀스를 여전히 따른다. 새 기능은 **그 시퀀스 위에 새 단계를 쌓아** 구현한다.

결과: 단계 수는 **증가만** 하고 감소하지 않는다. 앞으로도:

- TPM 기반 하드웨어 검증 → 새 단계 추가 가능
- 포스트 퀀텀 암호화 → 새 단계 추가 가능
- 11단계, 13단계로 늘어날 여지 있음

**사용자님의 직관 (앞선 메시지의 요약)이 정확히 맞았습니다**:
> *"작은것에서 큰것으로 점점 chain으로 로딩하는 구조를 취하였고, 호환성때문이고, MBR 하나만 쓰이면 알잘딱 시절에서 이제는 복잡한 OS 가 선형적으로 진화하는 과정에서 컴플렉서블하고 더 많은 하드웨어 업데이트를 반영하기 위해 10단계 체인까지 진화했다. 앞으로도 새로운 단계가 추가될 필요가 있으면 한 방에 잘되는 것보다 호환성을 유지하기 위해 11단계 13단계 추가될 가능성도 존재한다."*

이 직관은 **정답** 이다. 각 단계의 필연성은 위 Q&A 에서 확인된 대로 "이전 단계가 해결 못하는 문제가 있어서" 가 답.

### 9-5. Born2beRoot 의 10단계 체인 재해석 — 각 단계의 필연성

```
[단계 1] VirtualBox 펌웨어 (BIOS 에뮬레이션)
  ↓ 왜 필요? 하드웨어 자가진단(POST), 부팅 장치 선택. OS 는 아직 없음.
  ↓ 다음 단계가 해결 못하는 것: "어느 디스크에서 부팅할지" 를 아무도 모름.

[단계 2] MBR 의 boot.img (GRUB stage 1, 440 bytes)
  ↓ 왜 필요? BIOS 가 고정 위치 512 bytes 만 읽을 수 있음. 거기에 첫 부트로더를 넣어야 함.
  ↓ 한계: 너무 작아서 파일시스템 못 읽음.

[단계 3] core.img (GRUB stage 1.5, MBR-sda1 사이 빈 공간)
  ↓ 왜 필요? 파일시스템(ext2) 드라이버를 내장. 고정 위치라 stage 1 이 찾을 수 있음.
  ↓ 한계: 메뉴 시스템 등 복잡한 기능은 아직 없음.

[단계 4] normal.mod (GRUB stage 2, /boot 안의 파일)
  ↓ 왜 필요? stage 1.5 가 ext2 를 읽을 수 있게 해서 파일 이름으로 접근 가능. 메뉴 표시, 커널 선택.
  ↓ 한계: GRUB 은 OS 가 아님. 커널 로드 후 자기 역할 종료.

[단계 5] Linux 커널 (vmlinuz, /boot 에서 로드됨)
  ↓ 왜 필요? 하드웨어 드라이버, 프로세스 스케줄러, 시스템 콜 제공.
  ↓ 한계: 순수 커널은 암호화된 루트를 마운트 못 함.

[단계 6] initramfs (/boot 에서 로드됨)
  ↓ 왜 필요? 루트 마운트 전에 필요한 사전 작업 (LUKS, LVM, 네트워크 등) 을 임시 환경에서 수행.
  ↓ 한계: 어디까지나 임시 환경.

[단계 7] cryptsetup (initramfs 안에서 실행)
  ↓ 왜 필요? sda5 의 LUKS 암호화 해독. 사용자에게 암호 요청.
  ↓ 한계: 해독만 함. 볼륨 매니저는 별도.

[단계 8] LVM 활성화 (initramfs 안에서 실행)
  ↓ 왜 필요? LUKS 해독한 볼륨 위의 LVM VG 를 인식, LV 들을 /dev/mapper/ 아래에 노출.
  ↓ 한계: 마운트는 아직.

[단계 9] switch_root 로 / 마운트 (initramfs → 진짜 루트)
  ↓ 왜 필요? 임시 환경(initramfs) 에서 진짜 영구 환경(LV 의 ext4) 으로 전환.
  ↓ 한계: 아직 첫 사용자 프로세스 없음.

[단계 10] systemd 시작 (/sbin/init 실행)
  ↓ 왜 필요? /etc/fstab 기반으로 /home, /var, /tmp, /srv, /var/log 마운트. 
  ↓           SSH, UFW, cron 같은 서비스 시작. 로그인 프롬프트 띄움.
  ↓ 완료: 사용자 사용 가능 상태.
```

**각 단계는 "이전 단계가 못하는 일" 을 수행하고 "다음 단계가 처리할 일" 을 남긴다.** 이게 chain loading 의 필연성.

### 9-6. 10단계 한 장짜리 요약표

머리에 그려지지 않을 때 이 표 하나만 외우면 된다. **각 단계가 "할 수 있는 일" 과 "못하는 일 → 다음 단계로 넘기는 책임"** 으로 구성됨.

| # | 단계 이름 | 어디에 있나 | 이 단계가 **할 수 있는 일** | **못해서** 다음 단계 필요 |
|---|---|---|---|---|
| **1** | BIOS (펌웨어) | 메인보드 ROM | 하드웨어 자가진단 + 디스크 첫 섹터(512 B) 읽기 | **파일시스템 개념 없음** → MBR 필요 |
| **2** | MBR boot.img | sda 의 sector 0 (440 B) | "다음 섹터로 점프" | **너무 작아서 FS 드라이버 못 넣음** → core.img 필요 |
| **3** | core.img | MBR–sda1 사이 빈 공간 (~1 MB) | ext2 읽기, /boot 접근 | **메뉴·풍부한 기능 없음** → normal.mod 필요 |
| **4** | normal.mod (GRUB) | /boot/grub/i386-pc/ | 메뉴 표시, 커널 선택, vmlinuz + initramfs 로드 | **GRUB 은 OS 아님** → 커널 실행권 넘김 |
| **5** | Linux kernel | /boot/vmlinuz-* | 하드웨어 드라이버, 메모리·프로세스 관리 | **사용자 공간 프로그램 없어 LUKS 못 품** → initramfs 필요 |
| **6** | initramfs | /boot/initrd.img-* (RAM 에 풀림) | cryptsetup·lvm·mount 같은 도구 제공 | **암호 아직 안 물어봄, 볼륨 미활성** → 다음 단계들 필요 |
| **7** | cryptsetup (initramfs 안) | RAM 임시 / | LUKS 암호 받아 sda5 해독 → sda5_crypt 생성 | **볼륨 매니저 역할 아님** → lvm 필요 |
| **8** | lvm (initramfs 안) | RAM 임시 / | VG 활성화, LV 들을 /dev/mapper/ 에 노출 | **아직 마운트 안 함** → switch_root 필요 |
| **9** | switch_root | initramfs 의 /init | 진짜 /(ext4) 를 마운트 후 루트 전환 | **사용자 서비스 시작 안 함** → systemd 필요 |
| **10** | systemd | 진짜 / 안 /sbin/init | /etc/fstab 마운트, 서비스 기동, 로그인 프롬프트 | **완료** — 사용자 사용 가능 |

### 9-7. 10단계를 외우는 핵심 한 문장

> **"각 단계는 이전 단계가 수용 불가한 요구를 해결하고, 결과적으로 자신도 수용 불가한 한계를 만나 다음 단계가 필요해진다."**

외우는 법 — 단계를 **3 묶음** 으로 분할:

```
┌─── 하드웨어 → 부트로더 (단계 1–4) ─────┐
│  BIOS → MBR → core.img → GRUB          │  "파일 시스템을 읽게 되기까지"
│  (512 B → 440 B → ~1 MB → 수 MB)       │
└─────────────────────────────────────────┘

┌─── 커널 → 루트 준비 (단계 5–8) ────────┐
│  Kernel → initramfs → cryptsetup → lvm │  "진짜 루트를 열어주기까지"
│  (커널이 임시 환경에서 암호화/볼륨 해결)│
└─────────────────────────────────────────┘

┌─── 루트 진입 → 사용자 환경 (단계 9–10) ┐
│  switch_root → systemd                  │  "서비스 시작, 로그인 가능"
└─────────────────────────────────────────┘
```

**3묶음 이름**으로만 먼저 외우고, 각 묶음 안의 4-4-2 단계는 필요할 때 꺼내 쓰는 방식이 효율적.

### 9-8. 학생의 부팅 스토리 — 어디가 맞고 어디를 고쳐야 하나

실제 학생의 자가 서술을 놓고 각 단계를 평가·교정한다. 이 과정을 따라가면서 **자기 이해의 허점** 을 확인하라.

#### 학생의 원래 스토리 (요약 편집)

> *"bios : mbr 영역으로 점프 → mbr : grub 영역으로 점프 → **고정 grub**: 파일시스템을 해석할 수 있는 드라이브 파일이 있는 쪽으로 점프 → **grub 파일시스템**: 파일시스템 드라이버 파일을 해석하여 이해할 준비가 되었고, 실제 grub 동작으로 여러 드라이브나 tui, 입력장치 등등 이제는 선택의 영역 즉 사용자 인터랙션 프로그래밍 영역을 해석할 준비가 되어 그쪽으로 점프 → **선택의 영역**: 시간이 자동으로 지나면 선택이 알아서 되겠으나 옵션에 따라 선택지가 존재함. 선택한 쪽으로 점프 → 리눅스 커널 시스템으로 들어온 것: 커널 시스템이 LUKS 로 암호화 프로그래밍을 돌리고 있음. 사용자가 암호를 풀면 복호화 시스템이 동작 이후 적절하게 로드를 시도하여 마운트됨. 마운트가 완료된 후의 영역으로 점프 → 리눅스 시작"*

#### 평가표

| 학생 서술 | 정확도 | 교정 |
|---|---|---|
| "bios : mbr 영역으로 점프" | ✓ 맞음 | BIOS → sda sector 0 을 0x7C00 에 로드 → `JMP 0x0000:0x7C00` |
| "mbr : grub 영역으로 점프" | ✓ 맞음 (세분화 필요) | MBR 안의 `boot.img` → MBR–sda1 사이 고정 섹터의 `core.img` 로 점프 |
| "**고정 grub**: 드라이브 파일이 있는 쪽으로 점프" | ⚠️ 용어 부정확 | **"고정 grub"** 이라는 것은 없음. 정확한 이름: **"MBR–sda1 사이 고정 섹터 위치의 `core.img`"**. "고정" 의 의미 = **파일시스템 안이 아니라 절대 섹터 번호로 접근 가능한 위치**. 여기 든 core.img 는 **드라이버 파일이 따로 있는 게 아니라, 자기 안에 이미 ext2 드라이버를 내장** 하고 있음. |
| "**grub 파일시스템**: 드라이버 파일을 해석... 사용자 인터랙션 영역으로 점프" | ⚠️ 용어 부정확 | **"grub 파일시스템"** 은 존재하지 않음. 정확히는: **"core.img 가 자기 내장 ext2 드라이버로 sda1(/boot) 파티션을 읽음 → `/boot/grub/i386-pc/normal.mod` 및 기타 .mod 파일 로드"**. 이 시점부터가 GRUB 의 **풀 UI**. |
| "**선택의 영역**: 시간 경과 → 선택" | ⚠️ 별도 단계 아님 | "선택의 영역" 은 독립 단계가 **아님**. **GRUB stage 2 (`normal.mod`) 의 한 기능** 일 뿐. 메뉴 표시·타임아웃·사용자 선택은 전부 normal.mod 가 담당. stage 2 안에서 일어나는 **하나의 하위 이벤트**. |
| "리눅스 커널 시스템으로 들어옴" | ✓ 전환 맞음 | GRUB 이 `vmlinuz-*` (커널) + `initrd.img-*` (initramfs) 를 메모리에 로드 → 커널에게 실행 제어권 넘김 |
| "**커널 시스템이 LUKS 로 암호화 프로그래밍을 돌리고 있음**" | 🚨 **중대한 오해** | **커널 자체는 LUKS 를 풀지 못함!** 커널은 암호화 알고리즘을 모르고, 사용자 암호를 받는 대화 기능도 없음. 이 작업은 **initramfs** 라는 임시 환경에서 이루어짐. initramfs 안의 **`cryptsetup` 프로그램** 이 사용자에게 암호를 묻고, **`dm-crypt` 커널 모듈** 이 실제 복호화 담당. **두 계층이 협업**. |
| "사용자가 암호 풀면 복호화 시스템이 동작" | ⚠️ 주체 불분명 | "복호화 시스템" = **initramfs 의 cryptsetup (사용자 공간)** + **커널의 dm-crypt 모듈 (커널 공간)** 이 협업. 어셈블리 수준으로는 2 프로그램이 시스템 콜을 주고받으며 동작. |
| "로드를 시도하여 마운트됨" | ⚠️ 주체·순서 불분명 | 정확한 순서: (1) `lvm` 이 VG 활성화 → (2) `mount` 명령이 진짜 / 를 `/new_root` 에 마운트 → (3) `switch_root` 가 initramfs 의 / 에서 진짜 / 로 루트 전환. **3 개의 독립 프로그램** 이 순차 실행. |
| "마운트된 영역으로 점프 → 리눅스 시작" | ⚠️ 너무 뭉뚱그림 | "리눅스 시작" = **systemd(/sbin/init) 가 PID 1 로 실행** 되어: `/etc/fstab` 의 나머지 마운트 (`/home`, `/var`, `/tmp`, `/srv`, `/var/log`) 수행, **유닛 파일** (`.service`, `.target`) 처리, SSH·UFW·cron 등 서비스 시작, 최종적으로 **로그인 프롬프트** 표시. |

#### 정정된 전체 스토리

위 학생 서술을 올바른 용어로 고쳐 쓴 풀 버전:

```
[1] BIOS (VirtualBox 펌웨어) 시작
      POST → 부팅 디스크 선택 → sda sector 0 (512 B) 을 메모리 0x7C00 에 로드
      ↓ JMP 0x0000:0x7C00

[2] MBR boot.img (440 B) 실행
      "다음 섹터 위치로 점프" 외 다른 일 못함.
      → 하드코딩된 섹터 번호로 core.img 위치 찾기.
      ↓ JMP core.img

[3] core.img (MBR–sda1 사이 빈 공간 ~1 MB)
      자기 안에 내장된 ext2 드라이버 가동.
      → sda1(/boot) 파티션을 ext2 로 마운트하고 읽기 가능.
      → /boot/grub/i386-pc/normal.mod 및 모듈들 로드.
      ↓ JMP normal.mod

[4] GRUB stage 2 (normal.mod + 모듈들)
      메뉴 그리기, 키보드 대기, 타임아웃 진행 등.
      사용자 선택 (또는 타임아웃 기본값) → 해당 엔트리의 kernel/initrd 경로 읽기.
      → /boot/vmlinuz-* (커널) 과 /boot/initrd.img-* (initramfs) 를 메모리에 로드.
      → 커널에 제어권 넘김.
      ↓ kernel 진입

[5] Linux kernel 시작
      하드웨어 드라이버 초기화, 메모리 관리 자료구조 구축.
      메모리에 올라온 initramfs 를 "임시 /" 로 마운트.
      → initramfs 안의 /init 스크립트 실행.
      ↓ /init 실행

[6] initramfs 의 /init 스크립트 (임시 환경)
      ├─ dm-crypt 커널 모듈 로드 (modprobe dm-crypt)
      ├─ "Please unlock disk sda5_crypt:" 출력
      ├─ ← 사용자 암호 입력 대기
      ├─ cryptsetup luksOpen /dev/sda5 sda5_crypt  ← [7] LUKS 해독
      │    └─ 해독된 블록 디바이스 /dev/mapper/sda5_crypt 생성
      ├─ vgchange -ay                               ← [8] LVM VG 활성화
      │    └─ /dev/mapper/namykim42--vg-{root,swap_1,home,...} 나타남
      ├─ mount /dev/mapper/namykim42--vg-root /new_root  ← 진짜 / 를 임시 마운트
      └─ exec switch_root /new_root /sbin/init      ← [9] 루트 전환
           └─ 임시 initramfs / 버림, /new_root 가 새로운 / 가 됨
      ↓ systemd 진입

[10] systemd (= /sbin/init, PID 1)
      /etc/fstab 읽기 → 나머지 LV (/home, /var, /tmp, /srv, /var/log) 마운트.
      유닛 파일 (sshd.service, ufw.service, cron.service 등) 기동.
      로그인 프롬프트 (getty@tty1.service) 띄움.
      ↓ 완료
```

#### 학생이 놓친 핵심 3 가지

1. **initramfs 의 존재**: 커널과 진짜 / 사이에 **중간 계층** 이 있다. 커널 혼자는 LUKS 를 못 품. 사용자 공간의 cryptsetup 이 initramfs 에 들어 있어야 함.

2. **사용자 공간 vs 커널 공간의 협업**: "복호화 시스템" 은 하나가 아니라 **cryptsetup (사용자 공간) + dm-crypt (커널 공간)** 이 시스템 콜을 주고받으며 협업. 이건 OS 의 핵심 구조 (user/kernel 분리) 와 연결됨.

3. **"마운트" 의 주체가 여럿**: `lvm vgchange`, `mount`, `switch_root` 는 각각 독립 프로그램. 한 번의 "마운트" 가 아니라 **3 번의 단계** 가 있음. 특히 `switch_root` 는 루트를 교체하는 특별한 연산 — initramfs 의 / 에서 진짜 / 로 **루트 자체가 바뀜**.

#### 용어 정리 — 학생이 만든 비공식 표현 → 정식 이름

| 학생의 표현 | 정식 이름 |
|---|---|
| "고정 grub" | MBR 과 첫 파티션 사이 빈 공간에 저장된 GRUB **core.img** (= stage 1.5) |
| "grub 파일시스템" | 그런 건 없음. 정확히는 **"/boot 파티션(ext2) 안의 GRUB 모듈 파일들"** |
| "선택의 영역" | GRUB **stage 2** (normal.mod) 의 한 기능인 **"메뉴·타임아웃·사용자 선택"** 단계 |
| "커널이 LUKS 를 돌리고 있음" | 커널이 아니라 **initramfs 의 cryptsetup** 이 LUKS 처리. 커널은 dm-crypt 모듈로 암호화 계산만 담당 |
| "복호화 시스템" | **cryptsetup (사용자 공간)** + **dm-crypt (커널 공간)** 의 협업 |
| "마운트됨" | 실제로는 **lvm 활성화 → mount → switch_root** 의 3-단계 과정 |

이 용어들이 정착되면 이후 본문의 모든 부팅 관련 설명이 정확하게 읽힌다.

## §10. 학계급 부팅 모델 — 4 축 프레임워크 (Capstone)

> **이 장의 목적**: §1–§9 까지는 부팅 과정을 **선형 단계 (BIOS → MBR → ... → systemd)** 로 봤다. 이 시각은 학부 수준으로 충분하지만, **석사·연구급 직관** 을 만들려면 한 부팅 단계를 **4 가지 축으로 동시에** 볼 수 있어야 한다. 이 장은 §1–§9 의 내용을 새 시각에서 재구성하는 *capstone*.

### 10-1. 왜 선형 단계 모델만으로는 부족한가

5 개 대중 자료 (freeCodeCamp, Mansouri Medium, nu11secur1ty GitHub, GeeksforGeeks, Shafique Medium) 를 비교 분석한 결과:

- 전부 **"BIOS → MBR → GRUB → Kernel → Init"** 식의 **선형 handoff** 모델만 사용
- 어느 자료도 **"왜 그렇게 설계되었는가"** 를 깊이 다루지 않음
- 어느 자료도 **"다르게 설계할 수도 있었나"** 라는 트레이드오프 분석 없음
- **LUKS·LVM** 같은 Born2beRoot 핵심 컨셉은 거의 부재

선형 모델의 한계:
- "단계 N 다음에 단계 N+1 이 온다" 만 알고 그치게 됨
- **같은 부팅 과정을 다른 시각으로 보는 능력** 이 안 길러짐
- 새로운 환경 (UEFI Secure Boot, 컨테이너, 임베디드) 을 만나면 처음부터 학습해야 함

**석사급 직관 = 한 현상을 동시에 여러 모델로 볼 수 있는 능력.**

### 10-2. 5 개 대중 자료 비교 결과

| 소스 | 단계 수 | 특징 | 한계 |
|---|---|---|---|
| freeCodeCamp | 6 | 명확한 선형 모델 | "Runlevel" 용어 (SysV 시대, 현대 systemd 와 불일치) |
| Mansouri (Medium) | 5 | GRUB sub-staging, switch_root 언급 | LUKS·LVM 부재 |
| nu11secur1ty (GitHub) | 7 | 다이어그램, PID 1 명시 | Runlevel 잔존 |
| GeeksforGeeks | 5 | UEFI 명시, kernel phase 세부 | GRUB sub-staging 없음 |
| Shafique (Medium) | 7 | GRUB stage 1/1.5/2 분리, systemd | initramfs·LUKS 부재 |
| **우리 제0장 §9** | **10** | **LUKS·LVM·initramfs 전부 명시** | 일반화 부족 (UEFI 경로 누락) |

**핵심 관찰**: 우리가 §9 에서 만든 10 단계 모델이 대중 자료 평균 (6 단계) 보다 깊다. 차이의 4 단계 = `cryptsetup`, `lvm`, `switch_root`, **initramfs 의 명시적 분리** = Born2beRoot 의 LUKS+LVM 구성 특수성 때문.

### 10-3. 5 개 자료에 모두 빠진 8 가지 컨셉

석사급 직관에 필요한데 어느 대중 자료도 다루지 않은 것:

| 빠진 컨셉 | 왜 중요한가 |
|---|---|
| **권한(Privilege) 하강** | 부팅은 firmware → kernel → user 로 권한이 점진 위임되는 과정. 보안의 본질 |
| **추상화 레이어 추가** | 각 단계는 새 추상화 (block → file → process → service) 도입. OS 설계의 정수 |
| **닭-달걀 (chicken-and-egg) 문제** | 부팅의 본질 = bootstrap 의 본질. 자기 자신을 어떻게 초기화하는가 |
| **신뢰 체인 (chain of trust)** | Secure Boot, IMA, dm-verity 같은 현대 보안의 토대 |
| **실패 복구 (failure recovery)** | 각 단계 실패 시 복구 위치 (single-user, recovery shell, kernel panic) |
| **UEFI vs BIOS 분기** | UEFI 는 MBR/GRUB stage 1 자체가 없음. 완전 다른 부팅 모델 |
| **컨테이너 부팅 비교** | Docker/LXC 는 BIOS/GRUB/kernel 건너뛰고 init 부터 시작. 부팅 vs "init 만 실행" 의 차이 |
| **virtual vs physical 분기** | VirtualBox/KVM 의 BIOS 는 펌웨어가 아닌 호스트 프로세스 (에뮬레이션) |

### 10-4. 축 1 — 권한 (Privilege) 하강

x86 CPU 는 권한 레벨 (ring) 을 4 단계 (+2 개 추가 모드) 로 나눈다.

```
Ring -2: SMM (System Management Mode)
   - 펌웨어 영역. 가장 높은 권한.
   - OS 도 모르는 사이에 동작 (예: CPU 온도 관리)
       ↓
Ring -1: Hypervisor
   - VirtualBox / KVM / VMware 가 위치
   - VM 환경에서 호스트가 게스트 위에 군림
       ↓ (게스트로 진입)
Ring 0: Kernel space
   - Linux 커널 본체. 메모리 보호 무효화 가능, 모든 명령 사용 가능
       ↓ (시스템 콜 / 인터럽트로만 진입 / 종료)
Ring 3: User space
   - cryptsetup, lvm, mount, systemd, bash 등
   - 메모리 보호 적용, 특권 명령 금지
```

**부팅의 권한 하강 추적**:

| 단계 | 권한 레벨 | 의미 |
|---|---|---|
| 전원 ON | Ring -2 | CPU 가 reset vector 에서 펌웨어 코드 실행 |
| BIOS / UEFI | Ring -1 / -2 | 펌웨어가 하드웨어 초기화 |
| MBR / GRUB | Ring 0 (real mode) | 부트로더가 커널을 메모리에 올림 |
| Kernel 시작 | Ring 0 (long mode) | Linux 가 protected/long mode 로 전환 |
| **첫 user 프로세스 (PID 1)** | Ring 0 → Ring 3 | **systemd / init 가 시스템 콜로 user space 진입** |
| 모든 daemon · 사용자 프로그램 | Ring 3 | 메모리 보호 하에 실행 |

**핵심 통찰**: 부팅은 **"가장 높은 권한에서 시작해 점점 낮은 권한으로 위임"** 하는 과정. 이 위임 메커니즘 (system call, interrupt, exception) 이 OS 보안의 토대.

**Born2beRoot 적용**: `cryptsetup` 은 Ring 3 (사용자 공간) 에서 실행되고, 실제 복호화 연산은 Ring 0 (커널의 dm-crypt 모듈) 에서 수행. 이 둘이 **시스템 콜로 협업**. 권한 분리가 보안 모델의 핵심.

### 10-5. 축 2 — 추상화 추가 (Abstraction Layering)

부팅은 **추상화 레이어를 한 칸씩 쌓아 올리는** 과정. §9 의 chain loading 을 추상화 시각으로 재해석:

```
Layer 0: 전기 신호 (트랜지스터 ON/OFF)
   ↓ + machine code 추상화 (binary instruction)
Layer 1: CPU 명령어 (real mode 16-bit)
   ↓ + sector 추상화 (BIOS read sector API)
Layer 2: 디스크 블록 (BIOS read sector — sector 0)
   ↓ + file 추상화 (filesystem driver)
Layer 3: 파일시스템 (GRUB 의 ext2 드라이버)
   ↓ + virtual memory 추상화 (kernel MMU)
Layer 4: 가상 메모리 (kernel 의 페이지 테이블)
   ↓ + process 추상화 (kernel scheduler, fork/exec)
Layer 5: 프로세스 (PID 1 systemd, PID N daemons)
   ↓ + service 추상화 (systemd unit)
Layer 6: 시스템 서비스 (sshd.service, ufw.service 등)
   ↓ + session 추상화 (PAM, login)
Layer 7: 사용자 세션 (login shell, X11/Wayland)
```

**핵심 통찰**: 각 레이어는 **아래 레이어의 복잡성을 숨기고 단순한 인터페이스를 위 레이어에 제공**. 이 "추상화 사다리" 의 단계마다 새 추상화가 도입됨.

**Born2beRoot 의 LUKS+LVM 추상화 사다리** (제0장 §1 의 5 층 저장 스택의 부팅 시 동적 구축):

```
Layer 2: sda5 (raw block device, 21 GB 의 암호화된 비트들)
   ↓ + LUKS 추상화 도입 (cryptsetup luksOpen)
Layer 2.5: sda5_crypt (해독된 block device — 평문 비트)
   ↓ + LVM 추상화 도입 (vgchange -ay)
Layer 2.7: namykim42--vg-{root, swap_1, home, ...} (논리 볼륨들)
   ↓ + ext4 추상화 도입 (mount)
Layer 3: / 파일시스템 (POSIX 파일 인터페이스)
```

각 단계는 "block 의 의미" 를 한 번씩 더 추상화함. 이게 **제0장 §1 의 5 층 저장 스택 모델** 이 부팅 과정 전체로 일반화된 모습.

### 10-6. 축 3 — 신뢰 체인 (Chain of Trust)

각 부팅 단계는 다음 단계의 **무결성 (integrity)** 을 검증할 수 있어야 한다. 이게 안 되면 악성 코드가 중간 단계를 가로채는 공격 (bootkit, rootkit) 이 가능.

```
[Root of Trust] TPM (Trusted Platform Module) 또는 펌웨어 ROM
   - 위변조 불가능한 하드웨어 보안 칩
   - 모든 신뢰의 출발점
       ↓ verify signature (Secure Boot 활성)
[Stage 1] BIOS / UEFI firmware
   - 자기 무결성 자체 검증
       ↓ verify signature
[Stage 2] Bootloader (shim → GRUB)
   - shim = Microsoft 서명된 작은 부트로더, GRUB 검증 후 로드
       ↓ verify (kernel signature)
[Stage 3] Kernel + initramfs
   - 커널이 모듈 서명 검증 (CONFIG_MODULE_SIG)
       ↓ verify (dm-verity 활성 시)
[Stage 4] root filesystem
   - 파일시스템 자체의 해시 검증 (변조 감지)
       ↓ verify (IMA/EVM 활성 시)
[Stage 5] user processes
   - 실행 파일별 해시 검증
```

**핵심 통찰**: 신뢰 체인의 한 고리만 끊어져도 전체 체인이 무너짐. **부팅 보안 = 사슬의 가장 약한 고리**.

**Born2beRoot 와의 관계 — LUKS 는 신뢰 체인이 아니다**:

- Born2beRoot 는 **Secure Boot 비활성** (학습 목적)
- LUKS 는 신뢰 체인의 일부가 **아님** — LUKS 는 **기밀성 (confidentiality)** 메커니즘이지 무결성 (integrity) 이 아님
- 즉 LUKS 는 *"디스크를 훔쳐 가도 데이터를 못 본다"* 는 보장은 주지만, *"부팅 코드가 변조되지 않았다"* 는 보장은 안 줌
- 이 둘 (**기밀성 vs 무결성**) 의 구분이 보안 설계의 핵심
- 무결성 보장이 추가로 필요하면 **Secure Boot + dm-verity + IMA** 조합 필요

### 10-7. 축 4 — 자원 초기화 단계

부팅은 **각 자원 클래스가 순서대로 초기화** 되는 과정. 자원 의존성 관점:

```
Phase A: 하드웨어 검증 (POST)
  - CPU·메모리·디스크 자체 진단

Phase B: 디스크 접근 가능
  - BIOS sector read → GRUB FS driver 로딩

Phase C: 메모리 관리 활성화
  - kernel 의 페이지 테이블 구축, MMU 활성화

Phase D: 프로세스 모델 활성화
  - kernel scheduler, PID 0 (kernel idle), PID 1 (init) 생성

Phase E: 사용자 공간 (user space) 활성화
  - cryptsetup, lvm, mount 같은 사용자 프로그램 실행 가능

Phase F: 영구 스토리지 (persistent storage) 마운트
  - root filesystem 마운트 후 /usr, /var 등 접근 가능

Phase G: 시스템 서비스 활성화
  - systemd 가 daemon (sshd, cron, ufw) 기동

Phase H: 사용자 세션 (user session) 시작
  - getty/login → 로그인 프롬프트 → bash
```

**핵심 통찰**: 각 phase 는 **이전 phase 의 자원 위에서만 가능**. Phase E (사용자 공간) 는 Phase D (프로세스 모델) 없이 불가능. 이 의존성 그래프 = 부팅의 진짜 구조.

**Born2beRoot 의 initramfs 단계 = Phase E 와 Phase F 사이의 다리**:
- Phase E: 사용자 공간 가능 (initramfs 안의 cryptsetup 실행)
- 작업: cryptsetup → dm-crypt 모듈 → 해독 → lvm → mount
- Phase F: 진짜 / 마운트 완료
- 이 다리가 없으면 / 마운트 불가 → kernel panic

### 10-8. 4 축 종합 — 한 단계를 4 시각으로 동시에 보기

예제: **"initramfs 가 cryptsetup 으로 LUKS 를 해독하는 순간"**

이 한 순간을 4 축으로 동시에 분석:

| 축 | 분석 |
|---|---|
| **축 1: 권한** | Ring 3 (cryptsetup 사용자 공간) ↔ Ring 0 (dm-crypt 커널 모듈) 의 시스템 콜 협업. 권한 분리가 LUKS 보안 모델의 토대 |
| **축 2: 추상화** | block device (sda5, 암호화된 비트) → encrypted block device (sda5_crypt, 평문 비트) 추상화 추가. cryptsetup 이 새 추상화 레이어를 device-mapper 위에 박음 |
| **축 3: 신뢰** | "사용자 비밀번호" 라는 **인간 신뢰원** 이 신뢰 체인의 한 노드로 진입. 비밀번호가 곧 master key derivation 의 입력 |
| **축 4: 자원** | Phase E (사용자 공간) 에서 Phase F (영구 스토리지) 로 넘어가는 **다리**. 이 다리가 없으면 / 마운트 불가 → 부팅 실패 |

**4 축 모두 같은 "한 순간" 을 본다**. 하지만 각 축이 비추는 면이 다르다. 4 축 동시 시각 = **OS 의 한 행동을 시스템 설계 관점에서 입체적으로 이해** 하는 것.

**이 능력이 곧 컴퓨터 공학 석사급 직관**.

다른 단계 (예: GRUB stage 1.5 → stage 2 전환, switch_root 실행 순간) 에 동일한 4 축 분석을 적용해 보면 패턴이 굳어진다.

## §11. 학부 표준 정리 — *Operating System Concepts* §2.9.2 기준

> **메타·중요**: §1–§10 까지의 1,700 줄짜리 깊이는 **시스템 프로그래머·커널 개발자 진입 수준** 의 과잉 디테일이다. **학부 OS 강의 + 석사 입학 자격** 수준의 "정상 캘리브레이션" 은 표준 교재 **Silberschatz, Galvin, Gagne 의 *Operating System Concepts* §2.9.2 "System Boot"** 에 1.5 페이지로 끝나 있다.
>
> Born2beRoot 통과를 위한 **최소 필수 부팅 지식** 은 이 §11 한 장이면 충분. §1–§10 은 "더 알고 싶은 사람용 심화" 로 재포지셔닝.

### 11-1. 부팅의 정의 (교재 그대로)

> *"The process of starting a computer by loading the kernel is known as **booting** the system."*
> — Silberschatz et al., *Operating System Concepts*, §2.9.2

**부팅 = 커널을 메모리에 로드하여 시작하는 과정.** 한 문장으로 끝.

### 11-2. 부팅의 4 단계 (교재 표준)

> *"On most systems, the boot process proceeds as follows:*
> *1. A small piece of code known as the **bootstrap program** or **boot loader** locates the kernel.*
> *2. The kernel is loaded into memory and started.*
> *3. The kernel initializes hardware.*
> *4. The root file system is mounted."*
> — Silberschatz §2.9.2

| 단계 | 임무 |
|---|---|
| **1** | bootstrap program (= boot loader) 가 커널의 위치를 찾기 |
| **2** | 커널을 메모리에 로드하고 실행 |
| **3** | 커널이 하드웨어 (CPU, 메모리, 장치) 를 초기화 |
| **4** | root file system 마운트 |

이 4 단계가 **OS 교재가 부팅에 부여하는 정의의 전부**. 이걸 외우면 OS 시험 통과 가능.

### 11-3. 다단계 부팅 (multistage boot) — 보충

대부분의 시스템은 단번에 OS 를 로드하지 않고 여러 단계로 나눈다.

> *"When the computer is first powered on, a small boot loader located in nonvolatile firmware known as **BIOS** is run. This initial boot loader usually does nothing more than load a second boot loader, which is located at a fixed disk location called the **boot block**."*
> — Silberschatz §2.9.2

순서:

1. 전원 ON → **BIOS** (비휘발성 펌웨어 안의 작은 boot loader) 실행
2. BIOS 가 **두 번째 boot loader** 로드 (디스크의 고정 위치 = **boot block**)
3. boot block 의 프로그램은 보통 단순 (한 디스크 블록 안에 들어가야 함) → 나머지 부트스트랩의 디스크상 주소와 크기만 알고 있음
4. 그 부트스트랩이 OS 전체 로드

### 11-4. BIOS vs UEFI

> *"Many recent computer systems have replaced the BIOS-based boot process with **UEFI** (Unified Extensible Firmware Interface). UEFI has several advantages over BIOS, including better support for 64-bit systems and larger disks. Perhaps the greatest advantage is that UEFI is a single, complete boot manager and therefore is faster than the multistage BIOS boot process."*
> — Silberschatz §2.9.2

| | BIOS | UEFI |
|---|---|---|
| 구조 | 다단계 (multistage) | 단일 boot manager |
| 64-bit 지원 | 약함 | 강함 |
| 큰 디스크 | 2 TB 한계 (MBR) | 9.4 ZB (GPT) |
| 속도 | 느림 (단계 많음) | 빠름 |

**Born2beRoot 는 BIOS 부팅** (VirtualBox 기본값).

### 11-5. GRUB — Linux 의 부트로더

> *"GRUB is an open-source bootstrap program for Linux and UNIX systems. Boot parameters for the system are set in a GRUB configuration file, which is loaded at startup. GRUB is flexible and allows changes to be made at boot time, including modifying kernel parameters and even selecting among different kernels that can be booted."*
> — Silberschatz §2.9.2

특징:
- **오픈소스 부트로더** for Linux/UNIX
- 시작 시 **구성 파일** (`/boot/grub/grub.cfg`) 로드
- **부팅 시점 수정 가능** — 커널 파라미터 변경, 여러 커널 중 선택

**커널 파라미터 예시** (Silberschatz 가 든 예, `/proc/cmdline` 에서):
```
BOOT_IMAGE=/boot/vmlinuz-4.4.0-59-generic
root=UUID=5f2e2232-4e47-4fe8-ae94-45ea749a5c92
```

- `BOOT_IMAGE` : 메모리에 로드될 **커널 이미지 이름**
- `root` : **root file system 의 unique identifier** (UUID)

### 11-6. initramfs — 임시 RAM 파일시스템

> *"To save space as well as decrease boot time, the Linux kernel image is a compressed file that is extracted after it is loaded into memory. During the boot process, the boot loader typically creates a temporary RAM file system, known as **initramfs**. This file system contains necessary drivers and kernel modules that must be installed to support the **real** root file system (which is not in main memory). Once the kernel has started and the necessary drivers are installed, the kernel switches the root file system from the temporary RAM location to the appropriate root file system location. Finally, Linux creates the **systemd** process, the initial process in the system, and then starts other services."*
> — Silberschatz §2.9.2

순서:

```
1. 커널 이미지(vmlinuz) 는 압축 파일 → 메모리 로드 후 압축 해제
2. boot loader 가 임시 RAM 파일시스템 (= initramfs) 생성
   - 안에 real root 마운트에 필요한 드라이버 + 커널 모듈
3. 커널 시작 + 드라이버 설치
4. 커널이 root 를 (임시 RAM) → (real root) 로 switch
5. Linux 가 systemd (초기 프로세스) 생성
6. 다른 서비스 시작 (web server, DB 등)
7. 로그인 프롬프트 표시
```

### 11-7. recovery mode

> *"boot loaders for most operating systems—including Windows, Linux, and macOS, as well as both iOS and Android—provide booting into **recovery mode** or **single-user mode** for diagnosing hardware issues, fixing corrupt file systems, and even reinstalling the operating system."*
> — Silberschatz §2.9.2

대부분 OS 의 boot loader 가 제공:
- **recovery mode** : 하드웨어 진단
- **single-user mode** : 파일시스템 수리, OS 재설치

### 11-8. Born2beRoot 대입 — Silberschatz 4 단계가 우리 환경에서

| Silberschatz 단계 | Born2beRoot 실제 동작 |
|---|---|
| **1. boot loader 가 커널 위치 찾기** | BIOS → MBR → GRUB. GRUB 가 `/boot/vmlinuz-*` 위치 (sda1 ext2 안) 식별 |
| **2. 커널 메모리 로드·시작** | GRUB 가 `vmlinuz-*` + `initrd.img-*` 를 메모리에 로드, 커널에 제어권 넘김 |
| **3. 커널 하드웨어 초기화** | 디바이스 드라이버 초기화, MMU 활성화, dm-crypt·dm-mod 모듈 준비 |
| **4. root file system 마운트** | initramfs 의 cryptsetup → LUKS 해독 → lvm 활성화 → `mount /dev/mapper/...-vg-root` → `switch_root` → systemd 시작 |

## 제0장 자가 점검 체크리스트

- [ ] 1 byte 가 왜 8 bits 인지 역사적 이유를 말할 수 있다
	여러 사이즈 개념이 공존하던 시절이 있었으나, 8bit를 데이터 하나 단위로 보도록하는 시스템이 정착된 뒤로 이것이 고정되어지고, 추가로 더 확장된 데이터를 갖으려면 여러 byte를 합산하여 나름의 추상화된 데이터로 표현하자 라고 약속처럼 정착됨.
- [ ] 16진수 `0xAA` 를 10진수 170 으로 즉각 변환할 수 있다
	10*16 + 10 =  170
- [ ] `0x1BE` 가 446 인 이유를 계산으로 설명할 수 있다
	256*1 + 11*16 + 14 = 256+176+14 = 270 + 176 = 446
- [ ] little-endian 과 big-endian 에서 `0x12345678` 의 메모리 배치를 그릴 수 있다
	작은것부터 쓰는 형태, 얼만큼? 4바이트만큼 1바이트는 8bit 2진법으로 2자리, 16진법으로 0.5자리 따라서 2바이트는 1자리씩 따라서 4바이트는 2자리씩 따라서 78563412 라고 메모리에 배치됨.
- [ ] disk editor 의 `55 AA` 가 숫자값 `0xAA55` 인 이유를 설명할 수 있다
	리틀엔디안이란 가정하에 해당 데이터를 128비트 16바이트 데이터로 보겠다고 확정을 했다면 숫자로 복원하면 0xAA55가 되는거지요.
- [ ] **물리 섹터 / 논리 섹터 / block** 세 용어의 계층 관계를 설명할 수 있다
	물리 섹터: 물리적으로 1회 읽을때 최소 몇단위로 읽을 수 있는가? 기계적 기능 -> 현대 HDD는 4 * 1024 byte 즉 4KB로 읽는것에 최적화 되어 있다.
	논리 섹터: 호환성을 위하여 물리섹터가 얼만큼을 읽던지 간에 어셈블리어는 그냥 단순히 1섹터는 512 byte = 0.5kb 일거라고 생각하고 모든 하드웨어 동작을 정의해버린다. 이 중간과정에서 에러가 발생하지 않도록하는게 하드웨어 중간 컨트롤러의 역할이다.
	블록: 문맥마다, 사용자마다, 가장 애매하게 사용하는 단어 단위이다. 사전 지식이 없다면 오해하기 쉽상이다. 누군가는 하드웨어레벨에서 1블록이라 말할수도 있고, cpu 관점에서 1블록이라 말할수도 있고, 어셈블리, 메모리 등등 그냥 해당 카테고리의 최소컨토를 단위를 의미한다고 보면된다. 위키백과 MBR에서는 1블록을 512byte로 생각하고 말한것을 알수 있다.
- [ ] **LBA** 가 뭐고 `LBA 2048` 이 디스크의 어느 위치인지 계산할 수 있다
	Logical Block address: sentece ID 뭐 이렇게 표현해서 거대한 문서를 가지고 문장단위로 나눌때 id 단위로 나누는 경우가 빈번한데, LBA도 그냥 단위다. 여기서는 논리적 디스크 주소 단위라고 생각하면 되고, 0~511 까지 즉 512byte 단위로 넘어가다보니 LBA 2048의 시작점은 (2048 * 512)로 계산하면 된다 그냥. 따라서 직관적으로 편하게 1024 * 1024 이고, 1kb * 1024이니 1MB 부터가 LBA 2048이라고 생각할 수 있다. 
- [ ] **CHS 와 LBA** 의 차이 및 MBR 의 2 TB 한계가 왜 생기는지 수학적으로 설명할 수 있다
	CHS=Cylinder Head Sector: 과거 원기둥 형태의 하드디스크를 생각하면 된다. Cylinder는 몇번째 트랙인지, Head 는 어느 원판 어느면을 읽을지, Sector는 해당 트랙에서 몇번째 sector인지를 나타낸다. (c,h,s) 각각 숫자값으로 표현된다. 이걸 수학적으로 0x~ 으로 표현하면 하드디스크 물리적 주소가 나오는것이다.
	LBA는 디스크형태가 아닌 Memory SSD 시대에도 불구하고 이런 CHS와도 호환가능한가능한 규칙으로 사용가능하게 만드는 표준이라고 보면된다. 앞서 말했듯이 1블록은 512B 똑같고, 몇번째 번호인지에따라서 주소가 결정되어지고 당연히 중간 컨트롤러 필요하다.
	그래서 왜 2TB한계인가? MBR이 표현가능한 태생적 한계로 넘어가야되는데 총 64kb로 각각 16kb로 주소를 표현가능하고 .... 아 막힌다 명확하게 설명이 안된다. 애매하다 이거.
- [ ] `segment:offset` 표기를 보고 **물리 주소를 즉각 계산**할 수 있다
	segment:offset은 기본적으로 20bit로 주소값을 표현해야하는 한계를 16bit:16bit를 어떻게 해석할지 그 꼼수로 결과 값을 만드는데 앞에 16bit에 딱 16을 곱해버리면 그냥 2바이트만큼 비트연산이 일어나서 0xabcd(기존4자리)0 이 되어지고 이때 뒤에 offset만큼만 더하면 0x5자리로 32bit까지 표현이 가능한 주소값이 나오게 된다. 이때 오버플로우 위험이 있지 않냐고 하면 해당 오버플로우까지도 계산해서 더해줘야된다. 그게 몇인데? 왜 갑자기 20bit인데? 그거까지는 알아야될 범위는 아니라고 봤다. 
- [ ] `0x0000:0x7C00` 과 `0x07C0:0x0000` 이 같은 물리 주소인 이유를 계산으로 보여줄 수 있다
	0x00000 + 0x7c00 = 0x07c00
	0x07c00 + 0x0000 = 0x07c00
- [ ] **0x7C00 이라는 주소가 왜 32번째 KB의 첫 byte 인지** 를 `(32-1) × 0x400 = 0x7C00` 수식으로 증명할 수 있다
	0x400의 의미부터 알아보면 4*256 = 1024 즉 1KB이다. 32만큼 곱하면 32KB이다. 거기서 첫 32번째 KB의 첫 바이트란 의미는 31KB ~ 32KB 니까 31 * 0x400이고 0x7c00이 나온다.
- [ ] x86 CPU 가 전원 ON 후 왜 real mode 로 깨어나는지 말할 수 있다
	호환성 때문에 무조건 real mode로 작동하게 bios가 프로그래밍 되어 있다.
- [ ] **GRUB 이 왜 stage 1, 1.5, 2 세 단계로 나뉘는지** 각 단계의 존재 이유를 설명할 수 있다
	하드디스크가 너무나도 다양해졌다. OS가 너무 나도 다양해졌다. 그 OS를 불러올 프로그램이 다양해졌다. 따라서 이게 적절히 나눠져야되는데 단계를 설명은 못하겠다.
- [ ] **initramfs 가 왜 필요한지** Born2beRoot 맥락 (LUKS + LVM) 과 함께 설명할 수 있다
	뭔지 모르겠다.
- [ ] Born2beRoot 의 10단계 chain loading 을 각 단계의 "왜 필요한가" 와 함께 그릴 수 있다
	여전히 그게 머리에 그려지지 않는다.
